## 摘要

近年来, 以大语言模型 (Large Language Model, LLM) 为核心驱动的具身智能体, 在开放世界的自主决策与任务规划方面取得了显著进展。然而, 当面向 Minecraft 这类具有海量物品种类、深度合成依赖、长链路任务结构的开放世界沙盒环境时, 现有方案普遍存在三方面问题: 一是大语言模型在长链决策中容易出现"上下文遗忘"和"幻觉跳步", 难以稳定完成多阶段任务; 二是大多数智能体采用一次性把所有先验知识塞入提示词的被动方式, 信息冗余且 Token 消耗巨大; 三是任务执行流程缺乏严格的状态管理与失败恢复机制, 导致系统在意外错误下难以回退、复用和反思。

针对上述问题, 本文设计并实现了一个面向 Minecraft 的自主智能体系统。本文的主要工作和创新点包括:

1. **基于反思-行动 (ReAct) 范式的自主工具调用机制**。系统将知识检索、技能查询、世界观察、节点导航等能力统一抽象为可被语言模型调用的工具接口; 在每次调用前强制要求模型输出结构化的"思考字段"(thought), 形成"思考-检索-观察-再思考"的闭环, 显著降低了一次性提示词的体积与 Token 消耗。
2. **基于双有限状态机的协同流程控制架构**。系统由"任务规划状态机"与"动作执行状态机"两个并发协程协同工作, 通过异步消息总线进行通信, 将"规划-执行-验证-反馈-回退"的复杂逻辑显式化为可枚举、可调试的状态迁移图, 提升了系统的可解释性与稳定性。
3. **将长链决策链分解为拓扑有向无环图 (Topological DAG)**。系统以"状态节点"为图节点、以"动作转移"为有向边, 将一个高层目标 (例如"击败末影龙") 递归地分解为粒度逐级细化的有向无环图; 每个节点同时持有"预想状态快照"与"实际状态快照"两份信息, 实现了基于状态空间的可达性判定、节点级反思与失败回退。
4. **类海马体的离线经验巩固与长期记忆机制**。本文受生物海马体"睡眠期重放-皮层巩固"假说的启发, 在系统中实现了一个独立运行的离线记忆巩固模块。该模块在主决策循环空闲时对历史轨迹进行总结、抽取与索引, 自动生成可被未来任务检索复用的三类长期记忆: 程序性技能、失败教训与抽象任务模式, 形成"事实性知识 + 程序性知识 + 情景记忆"的三层认知架构。

实验部分以 Minecraft Java 版作为测试环境, 通过 Node.js 端的游戏交互框架与 Python 主程序进行 HTTP 通信, 在多组难度递增的任务上验证了系统的可行性。结果表明, 本文系统在任务成功率、Token 使用效率、长任务稳定性等指标上均显著优于以单一提示词直接驱动语言模型的基线方案; 经验巩固模块的引入进一步降低了重复任务的平均决策步数和 Token 消耗, 验证了"越用越好"的长期学习特性。

**关键词**: 大语言模型; 自主智能体; Minecraft; 反思-行动范式; 有限状态机; 有向无环图; 任务规划; 长期记忆

## Abstract

In recent years, embodied agents driven by Large Language Models (LLMs) have made significant progress in autonomous decision-making and task planning in open-world environments. However, when applied to sandbox environments such as Minecraft — which contain massive item spaces, deep crafting dependencies, and long-horizon task structures — existing approaches still suffer from three major issues: long-chain context forgetting and hallucinated step-skipping; passive "one-shot prompt" information injection that is highly redundant and token-expensive; and lack of explicit state management which prevents principled rollback, reuse, and reflection on failure.

To address these problems, this thesis designs and implements an autonomous LLM-driven agent system for Minecraft. The main contributions are as follows:

1. A **ReAct-style autonomous tool-invocation mechanism** that wraps the knowledge base, skill library, world observation and graph navigation as unified tool interfaces, allowing the LLM to retrieve information on demand via a self-reflective thought field, drastically reducing one-shot prompt size and token cost.
2. A **dual finite-state-machine cooperative control architecture**, in which a planning state machine and an execution state machine run concurrently and communicate through an asynchronous message bus, making the planning-execution-verification-rollback flow explicit, debuggable and stable.
3. A **topological DAG decomposition of long decision chains**, where each state node carries both an *imagined* and an *actual* world snapshot, enabling reachability checks, node-level self-reflection and principled rollback in state space.
4. A **hippocampus-inspired offline memory consolidation module**, which distils past trajectories into reusable long-term experience during the agent's idle time. The module automatically yields three categories of long-term memory — procedural skills, failure lessons and abstract task schemas — and forms a three-layer cognitive architecture together with factual knowledge (Wiki) and procedural skills.

Experiments on Minecraft Java Edition driven via a Node.js game-interaction layer show that the proposed system outperforms single-prompt LLM baselines in success rate, token efficiency and long-task stability. The memory consolidation module further reduces the average decision steps and token consumption of repeated tasks, validating the system's "learn-as-it-runs" property.

**Keywords**: Large Language Model; Autonomous Agent; Minecraft; ReAct; Finite State Machine; Directed Acyclic Graph; Task Planning; Long-term Memory

## 各章节内容思路简述

### 第1章 绪论

#### 1.1 研究背景

- 介绍开放世界游戏作为通用人工智能决策能力试金石的研究价值, 强调 Minecraft 作为"程序生成 + 长任务链 + 高自由度"基准环境的代表性, 引用 MineDojo、MineRL、Voyager 等已有研究。
- 阐述大语言模型从"问答模型"走向"决策智能体"的范式迁移, 总结当前 LLM 智能体在开放世界中面临的三大瓶颈: 长链遗忘、信息过载、缺乏稳定的执行框架与长期记忆。

#### 1.2 国内外研究现状

- **通用 LLM 智能体路线**: ReAct、Reflexion、Tree of Thoughts、AutoGPT、BabyAGI 等, 重点讨论它们在工具调用、反思与记忆上的得失。
- **Minecraft 智能体路线**: Voyager (终身学习 + 技能库)、DEPS、GITM、JARVIS-1、Plan4MC, 系统比较它们的任务表征、知识来源、长期记忆方式。
- **任务规划路线**: 经典分层任务网络 (HTN)、PDDL、行为树, 以及"LLM + 图搜索"的最新探索。
- **记忆与持续学习路线**: MemGPT、Generative Agents 中的记忆流、互补学习系统 (CLS) 在人工智能中的迁移。
- 本节末给出现状对比表 (任务表征 / 工具调用 / 状态控制 / 长期记忆), 引出本文创新点的位置。

#### 1.3 研究意义

本课题面向"大语言模型驱动的开放世界智能体"这一新兴方向, 以 Minecraft 为统一实验沙盒, 系统研究任务表征、决策架构与长期记忆三类核心问题。其研究意义可从理论、实践与工程方法论三个层面加以阐述。

##### 1.3.1 理论意义

**(1) 拓展"语言模型即决策核"的范式边界。** 传统大语言模型研究多聚焦于自然语言理解与生成, 而本课题将语言模型置于"感知—决策—执行—反思"的闭环中, 系统考察其作为通用决策内核的能力上限与失效模式, 为大模型从"问答系统"演化为"具身智能体"提供了可复现的实证支撑与方法论参照。

**(2) 引入有向无环图作为长任务的统一表征。** 现有工作多采用线性思维链 (Chain-of-Thought) 或单分支树状思维 (Tree of Thoughts) 表达任务结构, 难以同时承载"目标共享、路径多样、失败回退"三类需求。本文以"状态节点—状态转移边"形式化开放世界中的子目标空间, 论证了 DAG 在长任务链中相比线性与树状结构具有更高的可达性与鲁棒性, 为基于大模型的分层任务规划提供了新的形式化模型。

**(3) 借鉴互补学习系统假说构建人工记忆机制。** 本课题将认知神经科学中"海马体快速记忆—皮层慢速巩固"的双系统假说迁移到智能体设计, 提出"情景记忆—语义记忆—离线巩固"三级人工记忆模型, 为人工智能领域如何实现持续学习与经验沉淀给出了一种类脑启发的工程化解释, 丰富了类脑计算与认知架构在大模型时代的交叉研究内涵。

##### 1.3.2 实践应用价值

**(1) 推动开放世界游戏 AI 的研究进展。** Minecraft 作为程序生成、长任务链、高自由度并存的开放世界基准, 已成为衡量通用人工智能决策能力的重要试金石。本课题给出的智能体框架可直接服务于游戏 NPC 行为生成、关卡自动测试、玩家辅助代理等场景, 为游戏产业引入"会思考的非玩家角色"提供了一条可工程化的路径。

**(2) 可迁移至机器人长程任务规划。** 机器人长程操作 (Long-Horizon Manipulation) 与本文研究的虚拟世界任务规划在结构上高度同构: 二者皆面对部分可观测、动态变化、子目标可达性需反复验证的环境。本文提出的双状态机协同、想象—验证闭环与失败回退机制, 可直接迁移到家庭服务机器人、仓储分拣机器人、机械臂装配等具身智能任务, 缓解当前机器人系统"一旦失败即陷入死循环"的脆弱性。

**(3) 服务自动化运维与科研工作流编排。** 在 DevOps、数据流水线、生物信息分析等领域, 大量任务同样呈现"自然语言目标—子目标 DAG—工具调用执行—结果验证"的统一范式。本课题提出的"任务即配置"与"反思—行动—验证"工程模板, 可用于构建自动化运维代理、自动实验流水线、自动数据分析工作台等真实生产系统, 显著降低专家的重复劳动。

**(4) 为大模型应用降本增效。** 本文通过"按需检索"取代"上下文塞库", 通过"双状态机异步并发"取代"单线程长链路", 在保持任务成功率的同时显著降低了 Token 消耗与端到端时延, 对当下大模型推理成本高企的工业落地具有直接的经济价值。

##### 1.3.3 工程方法论价值

**(1) 提供一套可复用的智能体工程模板。** 本课题以依赖倒置与依赖注入容器为核心, 将语言模型供应商、知识源、执行环境、记忆策略等组件全部抽象为可替换接口, 形成分层清晰、可热插拔的开源框架。后续研究者可在不修改核心逻辑的前提下替换任意子模块, 从而加速智能体相关研究的迭代验证。

**(2) 形成智能体行为的可解释审计范式。** 通过强制思考字段、结构化最终答复协议与决策日志的全链路留痕, 智能体的每一次决策、每一次工具调用、每一次失败回退都成为可事后复盘的结构化证据。这一审计范式对大模型走向高风险、强监管的真实应用 (如医疗诊断、金融决策、自动驾驶) 具有重要的方法论参考意义。

综上, 本课题既在理论上探索了大语言模型作为开放世界决策内核的可行性边界, 又在实践上为多个具身智能与自动化领域提供了可直接迁移的工程模板, 兼具学术创新与产业落地双重价值。

#### 1.4 本文主要工作与创新点

围绕第 1.1—1.3 节所提出的研究背景与研究意义, 本文以 Minecraft 这一典型的开放世界沙盒环境作为统一实验载体, 设计并实现了一个由大语言模型驱动、具备长任务规划与持续学习能力的自主智能体系统——DreamCraft。本文的主要工作可概括为以下四方面:

**(1) 提出并实现了一个面向开放世界长任务的自主智能体整体框架。** 该框架以"领域模型—决策控制—认知服务—基础设施—具身环境"五层架构组织, 通过依赖倒置与依赖注入容器, 将语言模型供应商、知识源、执行环境与记忆策略统一抽象为可替换接口, 形成了一个可扩展、可热插拔的工程化基座。

**(2) 设计了一套以"状态节点—状态转移边"为核心的任务领域模型与图操作算法。** 该模型将长链决策过程显式化为可达的拓扑有向无环图, 支持分支扩展、路径注入、节点替换、剪枝与回退等运行时操作, 为后续的可行性检查、节点级想象、失败回退与多路径切换提供了统一的形式化基础。

**(3) 设计并实现了规划状态机与执行状态机协同的双有限状态机控制架构。** 二者通过基于主题路由的异步消息总线解耦通信, 配合"想象—执行—验证—回退"闭环与三次失败熔断机制, 使智能体的复杂决策流程显式化为可枚举、可追溯、可调试的状态迁移图。

**(4) 设计并实现了类海马体的离线经验巩固与长期记忆模块。** 系统以独立后台协程的形式运行经验蒸馏守护进程, 将历史轨迹自动蒸馏为"程序性技能 + 失败教训 + 抽象任务模式"三类结构化长期记忆, 并通过嵌入索引与近邻去重维护一个可持续增长的记忆库, 使智能体具备"会话间持续学习"的能力。

在上述工作的基础上, 本文形成的主要创新点如下 (与摘要严格对应, 4 项均已在系统中实现并完成实证):

**创新点 1: 基于反思—行动 (ReAct) 范式的自主工具调用机制。**

- *待解问题*: 传统智能体倾向于将整个知识库前置塞入提示词, 导致 Token 消耗巨大、信息冗余、关键信息被淹没, 且模型决策过程不可审计。
- *本文方法*: 将知识检索、技能查询、世界观察、节点导航等能力统一抽象为可被语言模型自主调用的工具接口, 并在每个工具的入参 schema 中强制嵌入"思考字段" (thought), 迫使模型在调用前显式输出推理过程; 同时引入结构化最终答复协议与解析失败回灌的格式自我修复机制, 形成稳定的"思考—检索—观察—再思考"闭环。
- *预期收益*: 相比单一提示词基线方案, 在保持任务成功率的同时显著降低 Token 消耗, 并将每一次决策的"为什么这么做"沉淀为可事后复盘的结构化日志, 使智能体行为具备可解释性。

**创新点 2: 基于双有限状态机的协同流程控制架构。**

- *待解问题*: 以 AutoGPT、BabyAGI 为代表的单循环智能体在长任务中难以同时兼顾"规划"与"执行"两类负载, 一旦中间步骤失败往往陷入混乱无法回退, 且整体流程不可枚举、不可调试。
- *本文方法*: 将"任务规划状态机" (8 态) 与"动作执行状态机" (5 态) 分别建模为两个并发协程, 通过支持精确等待 (`wait_for_topic`) 与非阻塞探取 (`fetch_topic`) 的异步消息总线进行通信; 规划侧负责可行性检查、状态想象、粒度判断、任务分解与路径导航, 执行侧负责代码生成、静态校验、异步下发与执行后验证, 两者可对不同节点并行推进。
- *预期收益*: 将"规划—执行—验证—反馈—回退"的复杂逻辑显式化为可枚举、可调试的状态迁移图, 在意外失败时由三次熔断机制自动触发路径回退而非崩溃, 显著提升长任务下的吞吐量与稳定性。

**创新点 3: 将长链决策分解为状态空间上的拓扑有向无环图。**

- *待解问题*: 思维链 (CoT) 与思维树 (ToT) 等表征只能表达线性或单分支的子目标序列, 无法同时刻画"目标共享、路径多样、失败回退"等真实任务中的关键结构, 也缺乏可被反复验证的状态语义。
- *本文方法*: 以"状态节点"为图节点、以"状态转移边"为有向边, 将一个高层目标递归地分解为粒度逐级细化的有向无环图; 每个节点同时持有"预想状态快照"与"实际状态快照"两份结构化信息 (含背包、装备、坐标、视野体素、生命/饥饿值、邻近实体等), 并通过"自底向上验证可达终点—不可达即剪枝"的初始化算法保证图始终为可达 DAG; 失败原因被记录于"被阻塞边集合"中, 由规划状态机重新选择替代路径。
- *预期收益*: 实现了基于状态空间的可达性判定、节点级反思与多路径替代切换, 使智能体在长链路任务下可以"目标共享、路径多样、失败回退", 显著降低重复扩展并提升整体可达性与鲁棒性。

**创新点 4: 类海马体的离线经验巩固与长期记忆机制。**

- *待解问题*: 现有大模型智能体多以"上下文窗口 = 全部记忆"的单层结构运作, 无法跨会话沉淀经验, 已踩过的坑往往重复踩, 已掌握的技能也难以稳定复用。
- *本文方法*: 受认知神经科学"海马体—皮层互补学习系统假说" (CLS, McClelland 1995) 启发, 在系统中引入与主决策循环异步解耦的离线巩固守护协程; 该协程在系统空闲时拉取情景缓冲中的成功/失败轨迹, 通过低成本语言模型蒸馏为"程序性技能 + 失败教训 + 抽象任务模式"三类结构化记忆, 并经嵌入索引、近邻去重合并、LFU 衰减遗忘后写入持久化记忆库; 主循环在每轮决策前以"状态快照 + 当前目标"为查询向量检索 top-k 长期记忆并注入提示词。
- *预期收益*: 形成"事实性知识 + 程序性知识 + 情景记忆"的三层认知架构, 使智能体在重复任务上的平均决策步数与 Token 消耗持续下降, 验证"越用越好"的长期学习特性, 同时为类脑认知架构与持续学习研究提供工程化样本。

四项创新点在系统中相互衔接: 创新点 1 解决"模型如何与外部世界交互", 创新点 2 解决"复杂流程如何被显式控制", 创新点 3 解决"长任务如何被结构化表征", 创新点 4 解决"经验如何跨会话沉淀", 共同构成一个闭环的、可持续演化的开放世界智能体框架。

### 第2章 相关技术与理论基础

> 写作思路: 本章不堆砌教科书内容, 只介绍后文系统设计与实现中真正依赖的关键技术与理论基础。每一节均以"概念简述—关键性质—在本文中的角色"三段式组织, 既为后续章节提供必要的术语铺垫, 又避免与正文功能重复。

#### 2.1 大语言模型与提示工程

**(1) Transformer 与解码器架构要点。** 当代主流大语言模型 (Large Language Model, LLM) 以 Vaswani 等人提出的 Transformer 为基础架构, 采用多头自注意力 (Multi-Head Self-Attention) 与前馈网络堆叠的解码器结构, 通过自回归方式逐 Token 生成输出。给定输入序列 $x_{1:t}$, 模型以条件概率 $P(x_{t+1} \mid x_{1:t})$ 预测下一 Token, 整段输出的概率分布可分解为:

$$P(y_{1:T} \mid x) = \prod_{t=1}^{T} P(y_t \mid x, y_{<t})$$

这一自回归特性意味着语言模型的"思考"本质上是一种概率链式生成, 一旦中间生成出现幻觉, 后续 Token 会沿错误路径继续衍生, 这也是后文需要通过 ReAct 范式与状态机控制对模型输出进行约束的根本原因。

**(2) 上下文窗口与 Token 计量。** 受位置编码与显存预算限制, LLM 的可处理上下文长度存在固定上界 (当前主流商业模型在数万至百万 Token 量级)。文本经字节对编码 (Byte-Pair Encoding, BPE) 或 SentencePiece 分词后转化为 Token 序列, 实际计费与时延均与 Token 数线性相关。本文系统涉及长链路任务, 单次任务对话往往可达数十轮工具调用, 因此 Token 经济性在设计层面具有一阶重要性, 直接驱动了第 4.3 节"按需检索式"提示词架构与第 4.6 节"上下文压缩"机制的设计。

**(3) 提示工程与提示词模板化。** 提示工程 (Prompt Engineering) 是指通过显式控制模型输入以引导其行为的工程实践。本文采用"角色模板 + 查询模板"的双层 Jinja 模板组合: 角色模板固化模型的身份与全局约束 (如"你是 Minecraft 智能体, 必须严格遵守工具调用格式"), 查询模板按任务类型注入运行时变量, 实现提示词与代码逻辑解耦, 便于调参与 A/B 比较。

**(4) 结构化输出约束。** 为让模型输出可被下游程序解析, 本文广泛采用 Pydantic 数据模型 + JSON Schema 的双重约束: 在工具调用层强制要求模型按预定义 schema 生成参数, 在最终答复阶段要求其遵循 `【Reason】 ... 【Final Answer】 ...` 结构化协议; 当解析失败时, 通过将异常以系统消息形式回灌到上下文, 形成"格式自我修复"机制。该机制是第 4.3 节自主工具调用稳定运行的关键。

#### 2.2 反思-行动 (ReAct) 范式与自主工具调用

**(1) 思维链与反思-行动的区别。** 思维链 (Chain-of-Thought, CoT) 通过显式输出中间推理步骤提升模型在复杂推理任务上的准确率, 但其推理过程完全发生在模型内部, 无法与外部世界交互。Yao 等人提出的反思-行动 (Reasoning + Acting, ReAct) 范式在 CoT 基础上引入"行动" (Action) 与"观察" (Observation) 两步, 形成可与外部环境闭环交互的"思考—行动—观察"循环:

$$\text{Trajectory} = (t_1, a_1, o_1, t_2, a_2, o_2, \dots, t_n, a_n, o_n)$$

其中 $t_i$ 为模型生成的思考 (Thought), $a_i$ 为基于思考决定的动作 (Action, 通常表现为一次工具调用), $o_i$ 为环境对该动作的反馈 (Observation)。本文将 ReAct 作为大模型与游戏世界、知识库交互的统一抽象基座。

**(2) 工具调用作为行动的具体形式。** 现代大语言模型 (如 GPT、通义、Claude 等) 普遍提供 "Function Calling / Tool Use" 接口, 允许在生成阶段直接输出符合 JSON Schema 约束的工具调用参数。本文将知识检索、技能查询、世界观察、节点导航、代码生成等能力统一封装为可被模型调用的工具, 并在每个工具入参 schema 中强制嵌入 `thought` 字段, 把"思考"显式纳入工具调用本身, 既保留了 ReAct 的可解释性又利用了模型原生的结构化调用通道。

**(3) 为何 ReAct 适合作为本文的工具调用基座。** 一是它天然支持长链路决策, 与本文 DAG 任务图结构契合; 二是它通过"思考"字段将模型的隐式推理外显化, 便于事后审计与失败溯源; 三是它通过"观察"环节引入外部反馈, 弥补了 LLM 静态知识截止的局限, 使智能体可基于实时状态做出动态决策。

#### 2.3 有限状态机与异步并发模型

**(1) Mealy 与 Moore 状态机。** 有限状态机 (Finite State Machine, FSM) 是一种由有限状态集合、输入字母表、状态转移函数与输出函数组成的离散计算模型, 形式化定义为五元组 $M = (Q, \Sigma, \delta, q_0, F)$, 其中 $Q$ 为状态集, $\Sigma$ 为输入符号集, $\delta: Q \times \Sigma \to Q$ 为状态转移函数, $q_0$ 为初始状态, $F$ 为终止状态集。Mealy 机的输出依赖于"当前状态 + 输入", Moore 机的输出只依赖于"当前状态"。本文第 4.2 节的双状态机均按 Moore 模型组织, 状态本身即决定了当前要执行的子任务 (如"可行性检查""状态想象""任务分解"等), 便于以状态名为索引绑定提示词与工具集。

**(2) 协程与事件循环。** Python 的 `asyncio` 库基于事件循环 (Event Loop) 调度协程 (Coroutine), 协程通过 `await` 主动让渡控制权, 由事件循环在 I/O 阻塞时切换到其他就绪任务, 从而以单线程实现高并发。本文系统中, 规划状态机、执行状态机、巩固守护协程均以独立协程的形式运行于同一事件循环, 避免了多线程同步开销, 同时获得了 I/O 密集型任务 (LLM 调用、HTTP 请求) 下的良好吞吐表现。

**(3) 异步生产者-消费者与邮箱-总线模型。** 当多个协程需要协同时, 通常以"消息队列"作为解耦中介, 形成生产者-消费者模式。本文进一步在普通队列之上构建了一个基于主题路由的"消息总线" (Message Bus): 每条消息携带主题与负载, 订阅方按主题精确等待 (`wait_for_topic`) 或非阻塞探取 (`fetch_topic`)。这一模型避免了传统轮询带来的 CPU 浪费与时延抖动, 使规划侧与执行侧在面对"等待对方完成某一具体事件"时具有原生的协程级语义支持。

**(4) 在本文中的角色。** 第 4.2 节的双状态机协同架构以本节的 Moore FSM 为骨架, 以协程为执行单元, 以主题消息总线为通信介质, 三者共同保证了"规划—执行—验证—回退"流程的可枚举性、可调试性与并发性。

#### 2.4 有向无环图与分层任务分解

**(1) 有向无环图的形式化定义。** 有向无环图 (Directed Acyclic Graph, DAG) 是一个二元组 $G = (V, E)$, 其中 $V$ 为节点集, $E \subseteq V \times V$ 为有向边集, 且不存在任何节点 $v$ 可经由若干条有向边回到自身。DAG 的关键性质包括: 拓扑排序的存在性 (节点可被线性化使所有边指向后继)、可达性闭包的可计算性、以及天然支持"多源—多汇—多路径"的层次结构表达。

**(2) 拓扑排序、路径搜索与剪枝。** 拓扑排序通常以 Kahn 算法或基于深度优先搜索的逆后序实现, 时间复杂度 $O(|V| + |E|)$。路径搜索可在拓扑序基础上以动态规划求得最短/最优路径; 剪枝则是在路径搜索过程中按一定可行性判据 (如不可达终点) 截断分支。本文第 4.1 节的"自底向上验证可达终点—不可达即剪枝"算法即是该范式的具体应用: 从终点反向标记可达节点, 凡未被标记者直接剪除, 保证图始终是可达 DAG。

**(3) 分层任务网络与本文任务图的关系。** 经典分层任务网络 (Hierarchical Task Network, HTN) 将复合任务递归分解为子任务直至原子动作, 与行为树 (Behavior Tree) 一道构成传统 AI 规划的两大范式。本文的任务图保留了 HTN 的"递归分解"思想, 但在表征上不再使用人工预定义的方法库 (method library), 而是由大语言模型在运行时动态生成子节点, 同时以 DAG 替代 HTN 的"任务树", 以容纳"多目标共享子任务"与"失败回退切换替代路径"两类传统 HTN 难以表达的结构。

**(4) "状态节点—状态转移边"形式化定义。** 本文将任务图节点定义为四元组 $n = (\text{name}, \text{desc}, s^\text{imag}, s^\text{real})$, 其中 $\text{name}, \text{desc}$ 为节点标识与自然语言描述, $s^\text{imag}$ 为规划阶段由模型生成的"预想状态快照", $s^\text{real}$ 为执行后采集的"实际状态快照"; 状态转移边定义为无序对 $\{n_i, n_j\}$ 上的有向操作, 携带"可执行代码""是否被阻塞""失败原因"等运行时元数据。该形式化是第 4.1 节算法描述的基础术语。

#### 2.5 检索增强生成与向量化知识库

**(1) 文本嵌入与向量空间。** 文本嵌入 (Text Embedding) 通过预训练编码器将自然语言映射为定长稠密向量 $\mathbf{v} \in \mathbb{R}^d$, 使语义相近的文本在向量空间中距离更近。本文采用与主流商业 API 兼容的 1024-1536 维嵌入模型, 对知识文档块、技能说明、长期记忆条目分别建立向量索引。

**(2) 距离度量。** 衡量两段文本相似度的常用度量包括欧氏距离 (L2) 与余弦相似度 (cosine):

$$d_{L2}(\mathbf{u}, \mathbf{v}) = \sqrt{\sum_{i=1}^{d}(u_i - v_i)^2}, \qquad \cos(\mathbf{u}, \mathbf{v}) = \frac{\mathbf{u} \cdot \mathbf{v}}{\|\mathbf{u}\| \, \|\mathbf{v}\|}$$

当嵌入向量经过 L2 归一化后, 余弦相似度与欧氏距离单调等价, 但余弦相似度对向量模长不敏感, 更适用于文本检索场景, 因此本文以余弦相似度为默认度量。

**(3) 近似最近邻索引。** 当向量库规模达到万级以上, 精确最近邻搜索 (Brute-Force kNN) 的时间复杂度 $O(N \cdot d)$ 难以满足实时检索需求, 业界普遍采用近似最近邻 (Approximate Nearest Neighbor, ANN) 索引, 例如 HNSW (Hierarchical Navigable Small World) 与 IVF-PQ (Inverted File + Product Quantization)。前者在多层导航小世界图上做贪心跳转, 检索时间近似 $O(\log N)$, 精度高、易增量更新; 后者通过聚类与乘积量化压缩向量, 适合超大规模。本文综合考虑增量写入需求与库容量, 选用 HNSW 作为长期记忆与知识库的统一索引。

**(4) 检索增强生成的必要性。** 检索增强生成 (Retrieval-Augmented Generation, RAG) 是指在生成前先检索相关上下文并注入提示词的范式。相比"把整个知识库前置塞入提示词"的朴素方案, RAG 在 Token 经济性、信息相关性、知识可更新性三方面均具有显著优势。这一点直接对应本文创新点 1 中"按需检索取代上下文塞库"的设计动机。

#### 2.6 互补学习系统与人工记忆巩固

**(1) 互补学习系统假说的生物学基础。** McClelland、McNaughton 与 O'Reilly 于 1995 年提出互补学习系统假说 (Complementary Learning Systems, CLS), 认为哺乳动物大脑通过"海马体—皮层"两套互补系统协同实现快速学习与长期巩固: 海马体负责快速、稀疏、特征独立的情景式记忆 (Episodic Memory), 在事件发生时迅速建立稀疏编码; 大脑皮层负责慢速、稠密、特征交叠的语义式记忆 (Semantic Memory), 通过反复学习抽象出泛化模式。睡眠或静息期间, 海马体中的短期痕迹通过"重放" (Replay) 缓慢迁移到皮层, 完成长期巩固并避免新旧知识相互干扰 (灾难性遗忘)。

**(2) 在人工智能中的对应物。** CLS 思想在深度学习领域演化出"经验回放" (Experience Replay)、双网络架构 (DQN 中的 online network 与 target network) 等机制。在大模型智能体领域, Generative Agents 中的"记忆流 + 重要性评分 + 反思"流程、MemGPT 中"主存—虚拟存"分层内存、Voyager 的"自我编程式技能库膨胀", 都可视为对 CLS 不同侧面的工程化模仿。

**(3) 情景记忆与语义记忆的工程化分离。** 在工程实践中, 情景记忆通常以原始轨迹/对话日志的形式存在, 检索粒度细但噪声大; 语义记忆则以结构化、去重、可索引的形式存在, 检索粒度粗但信噪比高。两者通过"蒸馏"流程相联: 情景记忆作为输入, 经语言模型抽取出结构化、可复用的语义记忆。本文第 4.7 节的离线巩固守护协程即是这一思想的具体实现。

**(4) 在本文中的角色。** 本节构成第 4.7 节"类海马体记忆模块"的理论铺垫: 主决策循环对应海马体的快速学习, 离线巩固协程对应睡眠期重放, 长期记忆库对应皮层语义网络, 三者通过事件驱动协作形成"事实性知识 + 程序性知识 + 情景记忆"的三层认知架构。

#### 2.7 Minecraft 仿真平台与具身交互层

**(1) Minecraft 与开放世界基准。** Minecraft 是一款基于体素 (Voxel) 的开放世界沙盒游戏, 其程序生成地形、深度合成依赖、长任务结构、多模态交互特性使其成为开放世界智能体研究的事实基准, 衍生出 MineRL (强化学习数据集与竞赛)、MineDojo (任务库与文本-视频知识库)、Voyager (终身学习智能体) 等代表性研究平台。本文采用 Minecraft Java 1.20.x 作为实验环境, 兼容主流社区机器人开发框架。

**(2) Minecraft Java 版协议与机器人开发框架。** Minecraft Java 版以 TCP 长连接承载二进制数据包, 服务器维护世界状态权威。Node.js 生态中的 Mineflayer 框架对协议进行了完整封装, 暴露出"挖掘""移动""合成""攻击"等高层动作 API, 并以事件回调形式上报世界变化。本文以 Mineflayer 为基础, 在其上构建一层 Express HTTP 服务, 将动作 API 与世界观察接口重新封装为语言无关的 REST 端点。

**(3) HTTP 服务作为 Python 与游戏世界的具身桥梁。** Python 主程序与 Node.js 端通过本地 HTTP 通信, 主要提供启动会话、世界观察、代码下发、关闭会话四类动作; 观察接口在 Node 端预先完成体素抽稀、实体距离归一化、背包序列化等预处理, 把原始 3D 世界投影为语言模型可读的结构化状态快照。这种"跨语言、跨进程"的桥接架构既复用了 Mineflayer 已有的协议实现, 又使主决策逻辑得以以 Python 异步生态独立演化, 二者通过明确的 REST 契约解耦。

**(4) 代码沙箱与正版账号支持。** 为安全执行语言模型生成的代码, Node 端将其包裹为立即执行的异步函数 (IIFE), 在 `vm` 模块下隔离运行, 配合超时与异常捕获形成沙盒边界; 同时基于 Microsoft / Mojang OAuth 设备流程实现正版账号登录, 由全局配置开关控制, 保证实验既可在离线模式快速迭代, 也可接入正版服务器做长期测试。该桥接层在第 4.5 节中将被进一步展开。

### 第3章 系统总体设计

> 写作思路: 本章在第 1—2 章对研究目标、相关技术与理论基础的铺垫之上, 自顶向下给出整个 DreamCraft 系统的全景视图, 重点回答"系统要解决什么问题—被分成了哪些层—层与层之间如何依赖—一次任务如何流动—底层用什么技术实现"五个问题, 不展开具体算法与代码细节 (后者见第 4 章)。

#### 3.1 系统目标与需求分析

##### 3.1.1 总体目标

DreamCraft 的总体目标是: 构建一个以大语言模型为决策核心、以 Minecraft 为具身环境的自主智能体系统, 使其能够仅凭一句自然语言目标 (如 "获得一把铁镐""击败末影龙"), 自主完成从任务理解、子目标分解、代码合成、具身执行、状态验证到经验沉淀的完整闭环, 并在多次会话间持续累积可复用经验。该目标既覆盖了"单次任务能完成"的可行性指标, 也强调了"长期使用越来越好"的持续学习特性。

##### 3.1.2 功能需求

按主用例划分, 系统需提供以下五类核心功能:

1. **自然语言任务接入**: 接受用户的自然语言目标作为唯一输入, 不假设其携带任何结构化字段, 由系统内部完成解析与子目标抽取。
2. **任务自主分解**: 在大语言模型协助下将高层目标递归分解为有向无环图形式的子目标空间, 并具备根据世界状态动态调整分解粒度的能力。
3. **具身自主执行**: 针对每个原子节点自动合成 JavaScript 动作代码, 经语法校验后下发到游戏环境异步执行, 并采集执行后的世界快照。
4. **执行结果自我验证**: 对比"预想状态快照"与"实际状态快照", 形成节点级反思; 失败时按既定策略回退路径而非直接终止。
5. **经验跨会话沉淀**: 在系统空闲时对历史轨迹进行结构化蒸馏, 生成程序性技能、失败教训与抽象任务模式三类长期记忆, 供后续任务复用。

##### 3.1.3 非功能需求

围绕开放世界长任务的工程特点, 系统在以下五个非功能维度提出明确指标:

1. **Token 经济性**: 优先采用按需检索而非全量上下文注入, 单次决策的平均 Token 消耗应显著低于"提示词塞库"基线。
2. **可解释性**: 每一次模型决策、工具调用与状态转移均留有结构化日志, 支持事后逐步回放与审计。
3. **可恢复性**: 任意单节点的执行失败不得导致系统崩溃; 系统应能依靠回退策略自动选择替代路径或终止上报失败原因。
4. **可扩展性**: 大语言模型供应商、知识来源、向量索引实现、执行环境等四类组件均可在不修改业务逻辑的前提下替换。
5. **长期可用性**: 系统在持续运行下应表现出"经验越积越多, 重复任务越做越快"的单调改进趋势, 而非性能漂移。

#### 3.2 总体架构设计

##### 3.2.1 分层架构概述

为同时满足上述功能与非功能需求, DreamCraft 采用自上而下五层架构: 领域模型层、决策控制层、认知服务层、基础设施层、环境层。每一层只依赖其下层提供的抽象接口, 上层对下层一无所知具体实现, 从而保证关键替换点的局部性。

- **领域模型层 (Domain Layer)**: 定义状态节点、状态转移边、任务图、状态快照、技能、长期记忆条目等核心领域对象。该层不依赖任何外部框架, 只用纯 Python 数据模型描述"系统在谈论什么"。
- **决策控制层 (Application/Core Layer)**: 承载规划状态机、执行状态机、异步消息总线、任务上下文管理器等运行时编排逻辑, 是整个系统的"中枢神经"。该层只调用认知服务层的抽象接口, 不直接接触具体的语言模型 SDK 或游戏协议。
- **认知服务层 (Service Layer)**: 提供四类高层服务——语言模型推理服务、知识检索服务、任务管理服务、记忆巩固服务, 对外呈现以"任务对象 → 结构化结果"为契约的统一调用入口, 内部聚合多个基础设施组件。
- **基础设施层 (Infrastructure Layer)**: 容纳具体的实现细节, 包括语言模型客户端 (OpenAI/通义兼容接口)、文本嵌入与向量索引、知识文档库、技能库仓储、提示词仓储、工具注册表、具身环境客户端等。该层是系统中唯一与外部 SDK / 第三方库直接交互的区域。
- **环境层 (Environment Layer)**: 系统外部的真实世界, 包括基于 Node.js 的游戏交互服务、Minecraft 服务器以及 Microsoft/Mojang 账号鉴权服务。系统通过 HTTP 接口与该层通信, 不在 Python 端假设任何游戏协议细节。

##### 3.2.2 依赖倒置与依赖注入容器

为保证上述"局部替换"特性, 系统在工程上严格执行依赖倒置原则 (Dependency Inversion Principle, DIP): 高层模块不依赖低层模块, 二者都依赖于抽象。具体到代码层面, 决策控制层与认知服务层仅持有协议 (Protocol) 接口, 不直接 import 任何具体实现; 实现类则在启动时由统一的依赖注入容器 (Container) 装配并交付。

容器负责: ① 在程序启动时按配置选择具体实现 (例如选择 OpenAI 还是通义嵌入); ② 维护单例与作用域 (例如向量索引在进程内仅初始化一次); ③ 在测试场景下注入模拟实现, 便于单元测试与回归验证。该容器以纯函数式风格组织, 不引入额外的运行时反射开销。

##### 3.3.3 替换点说明

分层架构的最大收益在于"局部替换": ① **大模型供应商可换**——只需在容器中替换 `LLMClient` 实现, 即可在 OpenAI、通义、Anthropic 等供应商之间切换, 业务代码无感知; ② **知识源可换**——通过替换知识仓储实现, 知识源可从本地 Markdown 切换至外部 Wiki API 或 RAG 即服务; ③ **执行环境可换**——只需重新实现具身环境客户端的协议接口, 即可把 Minecraft 替换为其他游戏甚至真实机器人; ④ **记忆策略可换**——记忆巩固服务的蒸馏策略、去重策略、衰减策略均可独立替换, 不影响主决策循环。这一可替换性正是第 1.3.3 节"工程方法论价值"中"可复用工程模板"的落地体现。

#### 3.4 数据流与控制流设计

##### 3.4.1 一次任务的端到端数据流

一次完整任务的数据在系统中的流动路径如下: 用户的自然语言目标进入决策控制层后, 首先被任务管理服务包装为根任务图节点 (起点 = 当前世界状态, 终点 = 目标状态); 规划状态机在大语言模型协助下不断对终点节点做可行性检查与状态想象, 并通过任务分解将其展开为子节点, 最终形成可达 DAG; 路径导航选定当前要执行的节点后, 执行状态机调用代码生成服务合成 JavaScript 代码; 代码经静态语法校验后通过具身环境客户端下发到 Node.js 服务并由 Minecraft 服务器执行; 执行完毕的世界快照沿原路径返回, 由执行状态机比对预想快照, 形成验证结果; 若成功则推进到下一节点, 若失败则将失败原因写入被阻塞边集合并触发路径回退。任务结束后, 完整轨迹被写入情景缓冲, 由离线巩固守护协程在空闲时蒸馏为长期记忆。

##### 3.4.2 三协程并发控制流

系统在控制流层面由三个独立的异步协程并发运行: **规划状态机协程**负责"思考下一步做什么"; **执行状态机协程**负责"把当前节点真正做完"; **巩固守护协程**负责"在系统空闲时整理过去的经验"。三者通过同一个基于主题路由的异步消息总线通信, 既保证彼此完全解耦, 又允许在需要时精确等待特定事件 (例如执行侧等待规划侧下发"开始执行 X 节点"的消息)。

这种"思考—执行—巩固"三协程并行的设计带来三方面收益: 一是规划侧可以在执行侧仍在执行当前节点时, 提前对后继节点做可行性检查与状态想象, 隐藏 LLM 推理时延; 二是巩固守护协程独立运行, 不会与主决策循环争抢 CPU 与上下文窗口; 三是任一协程发生异常都被局限在其状态机内部, 不会污染其他协程的状态机。

[图片占位 · **图 3-4 三协程控制流时序图** (fig_3_4_control_flow.png)]
*建议内容*: 绘制三条平行的协程时间轴——规划状态机 (Planning)、执行状态机 (Execution)、巩固守护 (Consolidator); 在时间轴之间用带主题标签的箭头表示消息总线上的通信事件 (如 `topic=execute_node`、`topic=node_done`、`topic=task_finished`); 重点展示"执行侧执行 N 时规划侧已在思考 N+1"以及"任务结束后巩固协程被唤醒"两类关键并发场景。

##### 3.4.3 失败回退与异常隔离

每个节点的执行状态机设有"三次失败熔断"机制: 同一节点连续失败超过 3 次时, 该节点对应的状态转移边被加入"被阻塞边集合", 控制权交还规划状态机重新选择路径; 当规划状态机发现当前节点无替代路径时, 进一步回退到其前驱节点重新分解。该机制保证了系统在面对模型幻觉、环境噪声、生成代码错误等多源不确定性时仍能稳定推进而非卡死。

#### 3.5 技术选型

##### 3.5.1 选型原则

技术选型围绕三条原则展开: ① **生态成熟**——优先选用社区活跃、文档完善的库, 降低维护成本; ② **可替换**——避免与特定厂商深度耦合, 关键能力 (LLM、嵌入、向量库) 均通过协议接口接入; ③ **异步友好**——所有 I/O 密集型组件均需提供原生异步 API 或可包装的异步接口, 以契合 asyncio 事件循环。

##### 3.5.2 主要技术栈

按层次列举系统采用的主要技术与选型理由:

| 层次 | 技术选型 | 主要理由 |
| --- | --- | --- |
| 主程序运行时 | Python 3.13 + asyncio | 主流异步生态、与 LLM SDK 兼容性最佳、Pydantic v2 性能提升显著 |
| 大语言模型接口 | OpenAI / 通义兼容 ChatCompletion API | 主流 SDK 接口统一, 便于跨供应商切换 |
| 数据模型与校验 | Pydantic v2 | 提供工具调用 Schema、配置校验、序列化能力, 同步支撑提示词与代码双侧约束 |
| 提示词管理 | Jinja2 模板 + Markdown 文件 | 模板与代码解耦, 便于版本化与 A/B 比较 |
| 文本嵌入与向量索引 | 商业嵌入 API + HNSW 索引 | 嵌入精度足够高且无需自建训练, HNSW 支持增量写入与高召回率 |
| 包管理 | Poetry | 锁文件可复现、虚拟环境管理统一 |
| 游戏交互层 | Node.js 20.x + Mineflayer + Express | Mineflayer 是 Minecraft 社区事实标准客户端, Express 暴露 REST 接口便于跨语言调用 |
| 代码静态校验 | JavaScript AST 解析器 (acorn 系) | 在代码下发前先做语法校验, 拦截 LLM 输出的语法错误代码 |
| 沙箱执行 | Node.js `vm` 模块 + 超时控制 | 与 Mineflayer 在同一进程, 同时获得隔离与低开销 |
| 鉴权 | Microsoft OAuth 设备流程 | 兼容正版 Minecraft 账号, 实验可在离线/正版双模式切换 |

##### 3.5.3 与既有方案的差异

与 Voyager 等以 LangChain 为统一中间件的方案不同, 本文未直接采用大型 Agent 框架, 而是仅借鉴其工具调用与 ReAct 的范式思想, 在核心层使用自研轻量协议接口与依赖注入容器。这一选择换来三方面优势: ① 避免被框架版本升级所绑架, 长期可维护性更好; ② 协议接口的语义更贴近本文领域模型 (任务图、状态节点、状态快照), 减少了概念转译成本; ③ 调试链路更短, 任何异常可直接在自有代码中复现, 不需深入框架内部。

### 第4章 关键模块的详细设计与实现 (论文核心章)

> 写作思路: 本章按照"领域模型 → 控制中枢 → 认知服务 → 基础设施"四级递进展开, 从最底层的数据抽象到最贴近用户的端到端体验, 逐节给出关键模块的设计动机、形式化定义、算法流程与工程实现要点。各节末尾保留与第 1.4 节四项创新点的对应说明, 形成"摘要—绪论—实现"一致的论述链。

#### 4.1 任务领域模型: 状态节点与状态空间图

##### 4.1.1 设计动机

开放世界任务的两大特征——"目标可达路径不唯一"与"中间步骤可能失败"——决定了任务表征不能采用线性链或单分支树, 必须支持节点共享、分支合并、失败回退与替代路径切换。为此, 本文以有向无环图 (DAG) 作为任务的核心表征, 并在节点层面引入"预想—实际"双轨快照, 使每一次状态转移都成为可验证的最小回路。

##### 4.1.2 状态快照: 世界状态的统一表征

状态快照 (State Snapshot) 是系统中描述"世界长什么样"的标准化数据结构, 也是可达性判定、状态想象与执行后验证三类决策任务的统一输入输出。每个快照按结构化字段组织以下信息: 背包内容 (物品 ID + 数量)、装备槽 (主手/副手/护甲)、当前坐标与朝向、视野范围内的体素列表 (类型 + 相对坐标, 已经过抽稀)、生命值与饥饿值、附近实体的类型与归一化距离, 以及一段自然语言"环境摘要"用于上下文注入。所有字段均通过 Pydantic 数据模型校验, 保证序列化前后无歧义。

##### 4.1.3 状态节点与状态转移边

**状态节点** (State Node) 定义为如下结构:

$$n = \langle \text{name}, \text{desc}, s^{\text{imag}}, s^{\text{real}}, \text{extra}, \text{Pred}(n), \text{Succ}(n) \rangle$$

其中 $\text{name}$ 与 $\text{desc}$ 分别为节点的短名与自然语言描述, $s^{\text{imag}}$ 与 $s^{\text{real}}$ 分别为规划阶段由语言模型生成的预想快照与执行后采集的实际快照, $\text{extra}$ 字段承载辅助信息 (如本节点对应的技能集合), $\text{Pred}(n)$ 与 $\text{Succ}(n)$ 维护节点的前驱与后继集合。

**状态转移边** (State Transition Edge) 定义为无序对上的有向操作:

$$e = \langle \{n_i, n_j\}, \text{direction}, \text{code}, \text{blocked}, \text{reason} \rangle$$

其中 $\{n_i, n_j\}$ 为基于哈希的无序对标识 (避免因方向不同重复存储), $\text{direction}$ 标记有向方向, $\text{code}$ 为该边上待执行的 JavaScript 动作代码, $\text{blocked}$ 与 $\text{reason}$ 分别记录该边是否被阻塞及阻塞原因。

##### 4.1.4 任务图与运行时上下文

**任务图** (Task Graph) 是系统对一次任务的最高层抽象, 维护以下运行时状态:

- 起点节点 $n_0$ (= 当前世界状态) 与终点节点 $n_T$ (= 目标状态);
- 节点列表 $V$ 与状态转移边列表 $E$;
- 当前执行路径 $\pi = (n_0, n_{i_1}, \dots, n_T)$, 由路径导航器在每次决策时更新;
- 被阻塞边集合 $B \subseteq E$, 用于实现失败回退;
- 执行历史 $H = (h_1, h_2, \dots)$, 记录每个节点的访问时间、成败、消耗 Token 等运行时元数据。

[图片占位 · **图 4-1 状态节点 / 状态转移边 / 任务图 UML 类图** (fig_4_1_domain_model.png)]
*建议内容*: 用 UML 类图绘制三类核心实体的字段与关系。`StateSnapshot` 在左, 列出背包/装备/坐标/视野体素/生命/邻近实体等字段; `StateNode` 在中, 关联两个 `StateSnapshot` (imagined / real); `StateEdge` 在右, 与 `StateNode` 多对多关联; `TaskGraph` 在顶部, 聚合 `StateNode` 列表、`StateEdge` 列表、阻塞边集合与执行历史。

##### 4.1.5 图操作算法

围绕任务图本文实现了一套图操作算法集合, 关键算法包括:

1. **分支扩展 (Branch Expansion)**: 在某节点处由语言模型生成多个候选子节点, 同时建立指向原后继的多条边, 实现"多路径并存"。
2. **节点替换式插入 (Substitutive Insertion)**: 当判定某节点粒度过粗需进一步分解时, 用一个子 DAG 替换该节点本身, 同时把原节点的前驱与后继重新接到子 DAG 的入口与出口。
3. **路径注入 / 路径链接**: 将一条由模型生成的子路径整体插入到现有图中, 并自动消解节点重名 (基于 name 哈希)。
4. **自底向上剪枝 (Bottom-up Pruning)**: 从终点反向遍历, 标记可达终点的节点; 凡未被标记者直接剪除。该算法保证任务图在任何时刻都是"可达 DAG"——即从起点存在至少一条路径到达终点, 否则规划状态机会立即上报失败。其伪代码如下:

```
Algorithm BottomUpPruning(G, n_start, n_end):
    reachable ← {n_end}
    queue ← [n_end]
    while queue is not empty:
        n ← queue.pop()
        for p in Pred(n):
            if (p, n) not in B and p not in reachable:
                reachable.add(p)
                queue.append(p)
    if n_start not in reachable:
        raise UnreachableGoal
    for n in V \ reachable:
        remove n from G
```

##### 4.1.6 可达性与失败回退

执行失败时, 失败节点对应的状态转移边被加入被阻塞边集合 $B$ 并截断当前路径; 路径导航器在剩余可达子图上重新搜索替代路径; 若已无替代路径, 则进一步回退到失败节点的前驱节点, 由规划状态机重新分解。该机制保证系统在多源不确定性 (模型幻觉、环境噪声、生成代码错误) 下仍能稳定推进, 而非陷入死循环或直接崩溃。

[图片占位 · **图 4-2 任务图示例: 以"获得铁镐"为例的 DAG 结构** (fig_4_2_task_graph_example.png)]
*建议内容*: 绘制一个真实任务图示例: 起点 = "初始状态", 终点 = "获得铁镐"; 中间节点呈 DAG 结构展开"砍木→木板→工作台→木镐→圆石→石镐→铁矿→熔炉→铁锭→铁镐", 同时在"铁矿"节点处分叉出"与村民交易换取铁锭"的替代路径, 体现"目标共享、路径多样"; 在执行中失败的边以红色虚线标注, 表示"被阻塞边"。

> **创新点 3 重点段落**: 之所以采用 DAG 而不是单链或树, 核心理由有四: ① **多路径并存**——同一目标 (如"获得铁镐") 既可通过自行采矿冶炼也可通过与村民交易, 单链无法表达; ② **节点共享子目标**——多个高层目标常依赖同一子目标 (如"工作台"既是制作木镐也是制作熔炉的前置), DAG 通过节点合并避免重复扩展; ③ **失败回退**——拓扑结构允许将失败的边加入阻塞集合后继续搜索替代路径, 而单链一旦中断只能整体重启; ④ **可验证最小回路**——节点上的"预想—实际"双轨快照使每条边的执行都成为可对比、可审计的最小验证单元, 这是节点级反思机制的形式化基础。

#### 4.2 双状态机协同控制: 规划状态机与执行状态机

##### 4.2.1 总体设计

为将"规划"与"执行"两类异质负载彻底解耦, 本文设计了两台并发运行的有限状态机: 规划状态机 (Planner FSM) 负责"思考下一步做什么"——可行性检查、状态想象、粒度判断、任务分解、路径导航; 执行状态机 (Executor FSM) 负责"把当前节点真正做完"——代码生成、静态校验、异步下发、执行后验证。两者通过基于主题路由的异步消息总线通信, 既保证彼此完全解耦, 又允许在必要时精确等待对方的特定事件。

##### 4.2.2 规划状态机 (8 态)

规划状态机的状态集合为 $Q_P = \{S_0, \text{Feas}, \text{Imag}, \text{Gran}, \text{Decomp}, \text{Nav}, \text{Wait}, \text{Done}\}$, 各状态语义如下:

- $S_0$ **初始化**: 接收用户的自然语言目标, 建立任务图的起点与终点节点, 转入可行性检查。
- $\text{Feas}$ **可行性检查**: 由语言模型基于当前世界状态与已掌握技能, 判定"从当前节点到下一节点是否可由原子动作直接达成"; 若可行转入状态想象, 否则转入任务分解。
- $\text{Imag}$ **状态想象**: 让语言模型预测下一节点完成后的世界状态, 写入该节点的预想快照 $s^{\text{imag}}$。该快照将在执行后被用作对照基准, 是节点级反思的核心机制。
- $\text{Gran}$ **粒度检查**: 判断节点是否已足够原子, 不再需要进一步分解; 是则进入路径导航, 否则进入任务分解。
- $\text{Decomp}$ **任务分解**: 在不可行/不够原子的节点处引入子目标, 通过节点替换式插入将其展开为一个子 DAG。
- $\text{Nav}$ **路径导航**: 在已展开的图中选择下一个尚未阻塞的可行替代路径, 经"自底向上剪枝"过滤无效分支。
- $\text{Wait}$ **等待执行**: 将选定节点广播到消息总线, 进入挂起态等待执行状态机的完成事件; 期间可异步预先对下一节点做可行性检查与状态想象, 实现"思考与执行并行"。
- $\text{Done}$ **任务完成**: 终点节点验证通过, 任务结束, 向情景缓冲写入完整轨迹。

[图片占位 · **图 4-3 规划状态机迁移图** (fig_4_3_planner_fsm.png)]
*建议内容*: 用圆形节点表示 8 个状态, 用带条件标签的箭头表示状态迁移; 突出 $\text{Feas}$→$\text{Imag}$→$\text{Gran}$→$\text{Nav}$→$\text{Wait}$ 主干流程, 以及 $\text{Feas}$→$\text{Decomp}$→$\text{Gran}$ 与 $\text{Gran}$→$\text{Decomp}$ 两条分解分支; 在 $\text{Wait}$ 与 $\text{Done}$ 之间标注"收到 node_done 且为终点节点"。

##### 4.2.3 执行状态机 (5 态)

执行状态机的状态集合为 $Q_E = \{E_0, \text{Idle}, \text{Gen}, \text{Verify}, \text{End}\}$:

- $E_0$ **初始化**: 启动后订阅消息总线上"开始执行某节点"的主题。
- $\text{Idle}$ **等待**: 阻塞等待新节点指令; 当收到节点指令后转入代码生成。
- $\text{Gen}$ **代码生成**: 由语言模型基于节点描述、当前状态快照与可用技能合成 JavaScript 动作代码, 经 AST 解析与依赖注入后通过具身环境客户端异步下发到 Node.js 服务执行。
- $\text{Verify}$ **验证**: 收集执行后的实际状态快照 $s^{\text{real}}$, 与预想快照 $s^{\text{imag}}$ 进行对比, 形成验证结果; 一致则向消息总线发布 `node_done` 主题, 不一致则触发代码再生成或熔断。
- $\text{End}$ **完成**: 通知规划侧并回到 $\text{Idle}$ 等待下一节点。

[图片占位 · **图 4-4 执行状态机迁移图** (fig_4_4_executor_fsm.png)]
*建议内容*: 用圆形节点绘制 5 个状态, 重点标注 $\text{Verify}$→$\text{Gen}$ 的"再生成"回边、$\text{Verify}$→$\text{End}$ 的"成功完成"边、以及 $\text{Gen}/\text{Verify}$→$\text{End}$ 的"三次失败熔断"虚线边。

##### 4.2.4 代码合成流水线与三次失败熔断

执行状态机的代码生成阶段并非简单地调用一次 LLM, 而是一条完整的合成流水线: **代码生成 → 依赖注入 → 静态语法校验 → 异步下发**。其中依赖注入会自动从技能库中拼接被引用的程序性技能函数, 避免模型重复生成; 静态语法校验调用 JavaScript AST 解析器, 在代码下发前拦截语法错误。

为应对持续性失败, 系统引入"三次失败熔断"策略: 同一节点连续失败次数 $c \geq 3$ 时, 该节点对应的状态转移边被加入被阻塞边集合 $B$, 控制权交还规划状态机重新选择路径或回退前驱。该阈值既给予模型足够的自我修复机会, 又防止无限重试导致 Token 与时间预算耗尽。

##### 4.2.5 异步消息总线

消息总线 (Message Bus) 是双状态机协同的通信介质, 基于 asyncio Future 与主题路由实现, 对外暴露三类原语:

- `publish(topic, payload)`: 向指定主题广播消息, 非阻塞返回。
- `wait_for_topic(topic)`: 在指定主题上挂起等待, 直到收到匹配消息后返回, 由协程 Future 提供精确通知, 无需轮询。
- `fetch_topic(topic)`: 非阻塞探取指定主题的最新消息, 用于"看一眼但不等待"的场景。

相比传统轮询或锁队列, 该模型在事件驱动协作下具有更低的时延抖动与更清晰的语义边界。

[图片占位 · **图 4-5 异步消息总线主题路由示意图** (fig_4_5_message_bus.png)]
*建议内容*: 中央绘制一个"消息总线"长条, 左右两侧分别绘制规划状态机与执行状态机两个圆角矩形, 中间用带主题标签的箭头表示三类典型消息流: `execute_node` (规划→执行)、`node_done` / `node_failed` (执行→规划)、`task_finished` (规划→巩固协程); 顶部用一个独立的小圆角矩形表示"巩固守护协程", 与总线虚线相连。

> **创新点 2 重点段落**: 与 AutoGPT、BabyAGI 等传统单循环智能体相比, "双状态机 + 异步总线"将"思考"和"执行"在控制流层面彻底解耦: 规划侧可以在执行侧执行当前节点的同时, 对下一节点做可行性检查与状态想象, 隐藏 LLM 推理时延; 任一状态机的异常被局限于其自身状态空间内, 不会污染另一方; 整个流程显式化为可枚举、可调试的状态迁移图, 极大提升了系统在长任务下的吞吐与稳定性。

#### 4.3 语言模型推理层与自主工具调用

##### 4.3.1 "任务即配置"的统一抽象

本文将每一类决策子任务 (可行性检查、状态想象、粒度检查、任务分解、路径导航、代码生成、验证) 都建模为一个结构化的"任务对象", 其字段包括: 唯一任务名、专属提示词模板、可用工具集合、最大迭代轮数、输出 Pydantic Schema、输出解析器与错误回灌策略。语言模型推理服务对外只暴露 `run(task, context) → result` 单一入口, 内部根据任务对象的元数据自动装配提示词、绑定工具集与解析器。这一"任务即配置"的抽象将决策控制层与具体提示工程彻底解耦, 上层只需在合适时机选择正确的任务对象, 即可获得结构化结果。

##### 4.3.2 强制思考字段与反思-行动主循环

为稳定 ReAct 范式的工程行为, 本文在每个工具的入参 schema 中强制嵌入一个 `thought: str` 字段, 迫使语言模型在调用工具前必须先输出一段自然语言推理。该字段同时承担两重作用: ① 作为模型的"内省日志", 提升其在长链路任务中的推理稳定性; ② 作为可事后审计的决策签名, 任何一次工具调用都可追溯到对应的思考内容, 形成完整的可解释审计轨迹。

整个反思-行动主循环按"思考字段 → 工具调用 → 工具执行 → 观察回灌 → 再思考"的节奏推进, 直到模型输出符合"最终答复协议"的终止标记为止。每一轮的输入与输出均被记录于结构化的迭代日志中, 与状态机状态、节点上下文一同沉淀, 构成系统的可观测基础。

[图片占位 · **图 4-6 反思-行动主循环时序图** (fig_4_6_react_loop.png)]
*建议内容*: 以时序图形式绘制三条泳道——LLM、工具集 (RAG / 技能查询 / 世界观察 / 节点导航 / 代码生成)、上下文管理器; 自上而下展示一轮典型 ReAct 迭代: LLM 输出 thought + tool_call, 工具集并发执行返回观察, 上下文管理器追加观察到对话历史; 最后一轮 LLM 输出 `【Reason】 ... 【Final Answer】 ...` 终止。

##### 4.3.3 并发工具调用

在单轮迭代中, 当语言模型同时给出多个互不依赖的工具调用 (例如同时调用语义检索与技能查询) 时, 推理服务以协程批量并发执行, 显著降低长链路总延迟。这一优化在主流 LLM SDK 的 `parallel_tool_calls` 接口之上自然落地, 无需对工具实现侧做任何改动。

##### 4.3.4 结构化最终答复协议与格式自我修复

为保证最终输出可被下游程序无歧义解析, 本文规定每个任务对象的最终答复必须以如下结构化协议收尾:

```
【Reason】
<自然语言解释>
【Final Answer】
<符合任务 Schema 的结构化结果>
```

解析器在收到回复后依次进行: 标记位完整性检查 → JSON/Schema 校验 → 业务级语义校验。任一环节失败, 解析器都会构造一条"系统消息"将异常信息回灌到模型上下文中, 提示其按指定格式重写。该"格式自我修复"机制使绝大多数轻微格式错误可在 1—2 轮内自动恢复, 显著降低任务因解析失败而中断的概率。

##### 4.3.5 提示词模板系统

提示词以"角色模板 (Role Prompt) + 查询模板 (Query Prompt)"的双层结构组织, 通过任务名作为键自动映射到对应的 Jinja 模板文件; 任务运行时由模板引擎注入运行时变量 (当前状态快照、节点描述、历史观察等)。模板与代码彻底分离, 便于版本化、A/B 比较与跨任务复用同一段角色描述。

> **创新点 1 重点段落**: 与"一次性把所有先验知识塞入提示词"的传统范式相比, 模型自主调用工具具有三方面优势: **(1) Token 经济性**——不再需要把整个知识库前置塞入上下文, 仅在模型显式表达"我还差什么"时才发起检索, Token 消耗显著降低 (见表 4-1); **(2) 信息相关性**——思考字段中的"我需要 X"通常直接对应一次精准检索, 检索结果即问题答案, 命中率高、噪声低; **(3) 可解释性**——思考字段是一份带签名的决策日志, 任何错误决策都可被定位到具体的一次工具调用与其前置思考, 便于事后复盘与定向修复。

#### 4.4 知识检索层: 语义检索、关键词检索与章节精读三级体系

##### 4.4.1 设计动机

事实性知识 (例如 Minecraft 物品配方、合成依赖、生物属性) 是任务规划与代码生成的重要输入。若一次性把全部知识塞入上下文将面临两难: 塞少则覆盖不足, 塞多则被关键信息淹没。本文借鉴人类"先模糊检索—再精读"的查阅习惯, 设计了三级渐进式检索体系。

##### 4.4.2 语义检索 (Semantic Search)

语义检索基于文本嵌入与 HNSW 近似最近邻索引, 接收自然语言查询并返回 top-k 候选文档块及其余弦相似度。返回结果中同时附带 L2 距离, 由语言模型自行评估检索质量并决定是否需要追加更精细的查询。该工具适用于"我大致知道想问什么, 但不确定关键词"的模糊场景。

##### 4.4.3 关键词全局检索 (Keyword Search)

关键词全局检索类似 ripgrep 的正则匹配, 直接在原始 Markdown 文档集合上进行字面匹配, 返回命中片段及其标题层级路径 (heading hierarchy)。该工具的核心价值在于让模型快速定位关键术语在哪个文档的哪个章节, 而非直接回答内容, 适合"我已经有明确关键词"的场景。

##### 4.4.4 章节精读 (Section Reader)

给定文件名与章节标题, 该工具抽取该章节及其所有子章节的完整 Markdown 内容并以原始格式返回。它通常作为前两级检索的下游, 用于"我已经知道答案在哪一节, 想深挖细节"的场景。三者构成"模糊 → 全局 → 精读"的逐级收敛检索流, 既保证召回率又控制单次返回内容的体积。

*建议内容*: 以漏斗形或阶梯形示意图展示从"语义检索 (模糊命中候选块) → 关键词全局检索 (定位章节) → 章节精读 (返回完整内容)"的递进过程, 每一级旁标注典型 Token 体积与适用场景。

##### 4.4.5 技能库: 程序性知识

技能库 (Skill Library) 是与事实性知识库并列的"程序性知识"组件, 存储已经经过验证的 JavaScript 函数及其自然语言摘要、参数说明、依赖关系。技能的写入有两个来源: 一是人工预置的基础技能 (移动、攻击、合成等), 二是第 4.7 节巩固守护协程从成功轨迹中蒸馏出的新技能。技能间的依赖关系通过 AST/正则解析自动维护, 在代码生成阶段被引用的技能函数会被自动拼接到生成代码之前, 避免模型重复生成已有函数。技能库与事实性知识库、长期记忆库一起构成"事实性知识 + 程序性知识 + 情景记忆"的三层认知架构。

#### 4.5 具身交互层: 游戏环境通信桥

##### 4.5.1 总体架构

具身交互层的职责是把 Python 主程序的决策结果"翻译"为 Minecraft 服务器可识别的协议动作, 并将 Minecraft 服务器的世界状态"翻译"为语言模型可读的结构化快照。整体采用"Python 端异步 HTTP 客户端 + Node.js 端 Express REST 服务"的跨进程架构, 二者以本地 HTTP 通信, 协议契约由共享的 JSON Schema 定义。

##### 4.5.2 Python 端客户端

Python 客户端基于 `httpx.AsyncClient` 实现, 对外暴露四个原语动作:

- `start_session(account, world)`: 启动一个游戏会话, 鉴权并加入指定世界。
- `observe_world(filter)`: 拉取当前世界的结构化快照, 可附带过滤器以减少返回体积。
- `dispatch_code(code, timeout)`: 将一段 JavaScript 代码下发到 Node.js 端异步执行, 同步返回执行 ID。
- `close_session()`: 优雅关闭会话并释放资源。

四个动作均为协程, 与决策控制层的状态机自然衔接。

##### 4.5.3 Node.js 端服务与观察预处理

Node.js 服务基于 Express 框架, 在 Mineflayer 客户端之上提供同名 REST 端点。其核心价值在于"世界观察"的预处理: 原始 Minecraft 视野中的体素可达数千个, 若直接序列化将爆掉 Token 预算, 因此服务端先做体素抽稀 (按类型聚合并保留最近 k 个)、实体距离归一化 (按欧氏距离排序并截断 top-N)、背包序列化 (按物品 ID 聚合)、坐标取整等预处理, 最终输出一份精炼的结构化 JSON 快照。

##### 4.5.4 代码沙箱

模型生成的代码并非直接 `eval`, 而是包裹在立即执行的异步函数 (IIFE) 中, 在 Node 端 `vm` 模块下隔离执行, 配合 300 秒超时与全局异常捕获形成沙箱边界。任何抛出的异常都会被结构化回报到 Python 端, 由执行状态机决定是触发再生成还是熔断。

##### 4.5.5 正版账号支持

为兼容正版 Minecraft 服务器, 系统基于 Microsoft OAuth 设备流程实现账号鉴权, 由全局配置开关在"离线快速迭代"与"正版长测"两种模式之间切换。鉴权过程异步进行, 不阻塞主决策循环。

#### 4.6 上下文压缩与 Token 控制

##### 4.6.1 长循环下的上下文膨胀问题

ReAct 主循环在长任务中可累积数十轮工具调用, 每一轮的工具回执都会留在对话历史中。若不加干预, 单次任务的对话历史很容易突破模型上下文窗口, 既产生截断风险, 又导致 Token 成本急剧上升。

##### 4.6.2 摘要工具与阈值触发

为此, 本文设计了"摘要工具 + 阈值触发"的会话内压缩机制: 当对话累计 Token 数超过预设阈值时, 系统自动调用一个独立的"摘要任务", 由低成本语言模型对当前对话历史生成一段结构化摘要 (包含已完成的子目标、关键观察、未解决的问题), 写入一段持久化的"全局摘要记忆", 并用该摘要替换早期工具回执消息, 仅保留最新一轮调用上下文。压缩前后, 模型的可见信息发生从"原始流水账"到"结构化摘要 + 最新调用"的迁移, 在保留关键决策依据的同时显著缩减 Token 体积。

##### 4.6.3 与长期记忆模块的概念关系

会话内压缩 (本节) 与会话间巩固 (第 4.7 节) 在概念上同源: 前者把"短期对话流水账"压缩为"本任务的全局摘要"以解决"上下文窗口爆炸", 后者把"已完成任务的完整轨迹"蒸馏为"跨任务长期记忆"以解决"经验跨会话沉淀"。二者共同构成 DreamCraft 的"压缩—蒸馏—复用"三级信息回收机制。

#### 4.7 类海马体的离线经验巩固与长期记忆模块 (创新点 4, 已实现)

> 本节描述本文已经设计并实现的离线记忆巩固模块。该模块作为一个独立的后台协程运行, 与主决策循环通过共享的任务上下文与异步事件解耦, 是创新点 4 的工程化落地。

##### 4.7.1 生物学动机与设计原则

**互补学习系统假说 (CLS, McClelland 1995)** 认为, 哺乳动物大脑通过两套互补系统协同实现学习: 海马体负责快速、稀疏、特征独立的情景式记忆 (Episodic Memory), 在事件发生时迅速建立稀疏编码; 大脑皮层负责慢速、稠密、特征交叠的语义式记忆 (Semantic Memory), 通过反复学习抽象出泛化模式。睡眠或静息期间的"重放"将海马体中的短期痕迹缓慢迁移到皮层, 实现长期巩固, 同时避免新旧知识相互干扰所引发的灾难性遗忘。

将该假说映射到本文系统, 可建立如下三重对应: 主决策循环 (ReAct) 对应海马体的快速学习; 离线巩固守护协程对应睡眠期重放; 长期记忆库对应皮层的语义网络。基于这一映射, 本模块遵循三条设计原则: ① 与决策主循环异步解耦, 不阻塞实时任务; ② 成功轨迹与失败轨迹分别处理, 避免相互污染; ③ 所有记忆条目高度结构化, 便于检索、去重与组合复用。

##### 4.7.2 模块组成

记忆模块由情景缓冲、巩固守护协程、记忆抽取器、长期记忆索引、去重遗忘机制、检索接入六个组件协同工作:

1. **情景缓冲 (Episodic Buffer)**: 主循环每完成一个任务图, 将完整轨迹写入情景缓冲, 包括任务图结构、状态快照序列、工具调用日志、生成代码、成败标签、Token 消耗等。该缓冲只承担短期暂存, 不参与检索。
2. **巩固守护协程 (Consolidator Daemon)**: 与主循环并发运行, 通过订阅"任务结束""任务失败"等主题在消息总线上被触发; 在系统空闲时, 拉取最近一段时间的情景片段, 调用低成本语言模型对成功/失败两条轨迹分别进行总结。
3. **记忆抽取器 (Distiller)**: 将每条轨迹蒸馏为以下三类结构化记忆条目:
   - **程序性技能 (Skill)** —— 可执行 JavaScript 函数及其语义摘要, 经语法校验后自动写入技能库, 完成"自我编程";
   - **失败教训 (Lesson)** —— 自然语言形式的反例, 包含触发条件、错误现象、规避建议, 在未来任务的可行性检查与代码生成阶段以"负样本"形式注入;
   - **抽象任务模式 (Schema)** —— 跨任务复用的子 DAG 模板 (如"探矿采集模式""熔炼合成模式""基础生存模式"), 在任务分解阶段被路径导航器优先匹配以降低决策成本。
4. **长期记忆索引**: 三类记忆均通过嵌入向量化建立索引, 与第 4.4 节的知识检索层共用同一套 HNSW 框架, 复用基础设施。
5. **去重与遗忘机制**: 每条候选记忆在写入前先进行近邻查询, 若与已有条目相似度高于阈值则触发"合并"而非"重复存储"; 同时基于使用频率维护一个 LFU 计数器, 极少被命中的条目会随时间衰减直至被覆盖, 模拟生物学上的遗忘曲线。
6. **检索接入**: 在主循环每轮调用语言模型之前, 检索器以"当前状态快照 + 当前目标"为查询向量, 拉取 top-k 长期记忆并注入到提示词的 `extra` 段, 使得模型在决策时立即获得"过去类似情况怎么做"的提示。

##### 4.7.4 工程实现要点

- 巩固协程独立运行于主事件循环, 通过消息总线接收"任务结束""任务失败"等订阅事件; 事件队列为空时进入低优先级休眠, 避免抢占主循环资源。
- 蒸馏调用使用低成本语言模型 (与主决策模型异源), 在保证蒸馏质量的同时进一步降低长期记忆的"摊薄"成本。
- 三类记忆采用统一的 Pydantic 数据模型, 通过自描述字段标识其类型、来源任务、置信度与最近命中时间, 检索时按类型与场景过滤。
- 记忆库支持磁盘持久化序列化, 跨次启动自动加载, 实现真正意义上的"会话间持续学习"; 同时支持手动导入/导出, 便于实验复现与跨实验共享。

##### 4.7.5 与已有工作的差异

- 与 Voyager 的"技能库自我膨胀"相比, 本模块不仅保存可执行技能, 还保存失败教训与抽象 DAG 模板, 形成"正例 + 反例 + 元模板"三位一体的长期记忆。
- 与 Generative Agents 中的"记忆流 + 重要性评分"相比, 本模块更强调"结构化蒸馏"而非原始日志检索, 检索结果可直接被决策环节使用, 无需二次解析。
- 与 MemGPT 的"操作系统式分层内存"相比, 本模块不依赖固定的页式调度, 而是通过事件驱动的异步巩固, 更贴近生物学上的"睡眠重放"机制。

> **创新点 4 重点段落**: 类海马体记忆模块的本质贡献是将"经验"从"一次性流水账"转变为"可跨会话沉淀的结构化资产"。技能层完成"自我编程", 让系统的可执行动作集合随时间单调增长; 教训层完成"自我警示", 让系统在重复相似错误前主动检查; 模式层完成"自我抽象", 让系统在面对结构相似的新任务时直接复用已验证的子 DAG 模板。三者共同实现了"越用越好"的长期学习特性, 也是本文区别于以单循环 LLM 智能体为代表的现有工作的核心。

### 第5章 系统测试与实验分析

#### 5.1 实验环境与基准任务

- **硬件/软件环境**: 开发与测试均在配置 macOS 与远程 Linux 服务器的环境下进行; Minecraft Java 1.20.x; Python 3.13; Node.js 20.x。
- **基准任务集** (按难度递增, 参考 Voyager 与 MineDojo 的任务设计):
  - L0: 砍木头 → 木板 → 工作台
  - L1: 木镐 → 圆石 → 石镐
  - L2: 铁矿采集 → 熔炼 → 铁镐
  - L3: 钻石镐 → 黑曜石 → 下界传送门
  - L4: 长任务"建立基础生存基地"(用于结构验证而非最终成功率指标)
  - L5: 击败末影龙 (用于验证长链路 DAG 可行性, 不作为主要量化指标)

#### 5.2 评价指标

- **任务成功率** (5 次重复取均值并报告标准差)
- **平均 Token 消耗** (按总量/缓存命中量/未缓存量分别统计)
- **平均决策步数** (规划状态机与执行状态机的总步数)
- **失败回退次数** (被阻塞边数量 + 三次熔断触发数)
- **代码一次通过率** (执行验证通过且无重试的比例)
- **首次任务 vs 复述任务** 的指标对比 (用于评估长期记忆收益)

#### 5.3 对比实验

- **基线 1**: 单一提示词全量注入 (无工具调用)
- **基线 2**: 反思-行动循环但无 DAG, 仅单链规划
- **基线 3**: 反思-行动 + DAG, 但单循环 (无双状态机)
- **基线 4**: 双状态机 + DAG, 但无长期记忆
- **本文完整方法**: 双状态机 + DAG + 长期记忆

> 用表格展示各组方法在 L0-L3 上的指标对比, 用折线图展示 Token 消耗随任务复杂度的增长曲线 (本文方法预期增长更平缓)。

#### 5.4 消融实验

- 去掉"想象-验证双轨快照" → 观察验证误判率上升、错误传播次数增加。
- 去掉"上下文压缩" → 观察长任务 Token 爆炸与上下文截断错误。
- 去掉"思考字段强制约束" → 观察模型跳过推理直接乱调工具的频率。
- 去掉"可行性预检查" → 观察无效任务分解次数。
- 去掉"代码静态语法校验" → 观察执行端运行时崩溃次数。

#### 5.5 长期记忆模块专项实验

> 该节面向创新点 4, 单独设计实验以量化记忆模块的实际收益。

- **实验 5.5.1 重复任务收敛实验**: 同一目标 (例如"获取铁镐") 重复执行 10 次, 记录每次的 Token 消耗、决策步数和成功率。预期: 第 1 次到第 10 次的 Token 消耗呈递减趋势, 决策步数收敛到原子级别, 成功率稳定在高位。指标: 收敛轮数、收敛后 Token 较首轮的下降比。
- **实验 5.5.2 跨任务迁移实验**: 在系统对 L1 (石镐链) 学习完毕后, 直接执行 L2 (铁镐链), 比较有无长期记忆的两种条件下 L2 的首次执行成本。预期: 经验启动的智能体能复用"采集 → 升级工具"的抽象任务模式, 显著缩短任务分解时间。
- **实验 5.5.3 失败教训防御实验**: 人为构造一类必然失败场景 (如"夜晚无光下挖矿被怪物击败"), 让系统先经历一次失败; 然后在另一个相似但不完全相同的场景下重新触发同类风险。预期: 经过巩固的智能体能在可行性检查阶段直接识别出风险并提前补足装备, 一次失败教训能在后续多任务中持续生效。
- **实验 5.5.4 记忆库规模与检索性能**: 将巩固后记忆库的条目数从 10 扩展到 1000, 测量检索延迟与命中质量, 验证近似最近邻索引在该规模下的可用性。
- **实验 5.5.5 去重机制有效性**: 关闭/开启近邻去重, 观察长时间运行后记忆库的冗余度与检索精度变化, 验证遗忘机制在抑制冗余上的作用。
- **实验 5.5.6 蒸馏粒度对比**: 对比"仅保存技能""仅保存教训""三类全开"三种配置下的任务表现, 验证三类记忆对最终性能的贡献并非可替代。

> 预期结论: 长期记忆模块在重复任务上能将平均 Token 消耗降低 30%-50%, 在跨任务迁移上能将首次任务成本降低 15%-30%, 且失败教训具有跨场景持续防御能力。

### 第6章 总结与展望

#### 6.1 工作总结

- 简述本文完成的工作: 从 0 到 1 实现了面向 Minecraft 的自主智能体系统, 在多组任务上达到预期效果, 并通过类海马体的离线巩固机制实现了"边运行边学习"的长期记忆能力。

#### 6.2 创新点回顾

- 重申 4 个创新点 (反思-行动自主工具调用 / 双状态机协同 / 状态空间 DAG 分解 / 类海马体离线记忆), 4 项均已在系统中实现并完成实证。

#### 6.3 不足之处

- 视觉感知缺位, 当前仅依赖结构化体素列表, 未引入多模态。
- 单智能体实验, 未涉及多智能体协作。
- 长期记忆的遗忘曲线参数仍为静态阈值, 缺乏自适应机制。
- 对极长链路任务 (击败末影龙) 仍依赖人工设定中间锚点。

#### 6.4 未来工作

1. 接入视觉模型 (CLIP / Llava / Qwen-VL) 作为辅助状态快照通道。
2. 多智能体协同 (建造、战斗、贸易) 与共享记忆库。
3. 用强化学习对反思-行动中的工具选择策略做进一步优化。
4. 将本文系统抽象为通用 LLM 智能体框架, 适配机器人、运维、办公自动化等任务。
5. 探索基于神经符号的更细粒度的记忆蒸馏方法, 替代当前以语言模型蒸馏为核心的实现。
