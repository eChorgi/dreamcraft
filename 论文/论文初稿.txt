# 摘要: 构建适用于Minecraft等开放世界的大语言模型具身智能体，目前仍面临上下文遗忘、Token消耗过高以及长链路任务容错率低等挑战。本文提出了一种全新的自主智能体框架，专为解决复杂环境下的长程决策而设计。该系统包含四个核心组件：1）基于ReAct范式的动态工具调用机制，大幅降低Prompt冗余与消耗；2）协同任务规划与动作执行的双有限状态机（FSM）架构，实现稳健的错误回退；3）拓扑有向无环图（DAG），将宏大目标分解为具备状态校验的细粒度子任务；4）受海马体启发的离线记忆巩固模块，从历史轨迹中持续提取程序性技能与失败教训。通过将复杂逻辑解耦为明确的状态迁移，该系统有效缓解了模型幻觉和执行崩溃。实验表明，我们的智能体在任务成功率、长程稳定性和Token效率上均显著优于传统的单Prompt基线方案。此外，长期记忆模块赋予了系统“越用越好”的终身学习能力，大幅降低了重复任务的决策步数与Token消耗。
## 1.1 研究背景
在通用人工智能（Artificial General Intelligence, AGI）的发展历程中，研究范式与评估基准的选择决定了智能体能力演进的上限。历史上，自主智能体在诸如雅达利（Atari）街机游戏、国际象棋以及围棋（Go）等特定领域取得了令人瞩目的突破，展现出了超越人类专家的决策水平 。然而，这些系统在本质上仍然属于“领域专家模型”。它们通常在高度孤立、状态空间有限且规则边界绝对明确的环境中，通过白板学习（Tabula Rasa）和人工设计的密集奖励函数进行训练 。这种过度简化的训练环境导致智能体极度缺乏泛化能力，一旦面对未知的任务分布或环境微调，其决策逻辑便会迅速崩溃。
人类智能的核心特征在于能够在一个充满无限可能、高度非结构化且动态演变的现实世界中，通过持续的终身学习（Lifelong Learning）来适应新任务并积累新技能 。因此，要孕育出真正具备人类认知水平的通用智能体，研究界必须构建一种“三位一体”的实验基准：支持多任务与多目标的复杂物理或虚拟环境、大规模的多模态先验知识库，以及具备高度灵活性与可扩展性的认知架构 。在这一背景下，以《我的世界》（Minecraft）为代表的开放世界沙盒游戏，无可争议地成为了当前大语言模型（Large Language Models, LLMs）和具身智能（Embodied AI）最理想的测试与演进环境。Minecraft 被学术界广泛认定为 AGI 研究的“微观宇宙”，其不仅是一个第一人称视角的 3D 虚拟空间，更是一个融合了连续控制、高维函数逼近和长期规划的复杂动态系统 。该环境完美复刻了现实世界决策问题的三大核心挑战：
第一，程序生成（Procedural Generation）带来的无限状态空间。与依赖固定地图和预设关卡的游戏不同，Minecraft 的世界是由算法通过程序生成的，每一次实例化都会创造出由不同化合物方块、地形地貌（如沙漠、森林、山脉）和生态群落组成的独一无二的环境 。这种机制从根本上杜绝了智能体通过“死记硬背（Rote Memorization）”地图坐标来完成任务的可能性。在程序生成的环境中，基于传统规则的智能体倾向于生成同质化的功能性行为，而更高级的智能体必须学会提取普遍适用的环境交互逻辑，应对“冷启动（Clean-state）”下的未知元素 。
第二，复杂的科技树与长任务链（Long Task Chains）。Minecraft 内置了极具深度的科技树和物品合成系统。智能体在游戏中的任何高级目标都需要被分解为极其漫长且有着严格依赖关系的子任务序列 。以“获取钻石”这一经典挑战为例，智能体必须经历一个跨度极长的时间周期：从寻找并砍伐树木、制作木板与工作台、合成木镐、开采圆石、升级石质工具、冶炼铁矿，一直到深入地下寻找特定的矿脉分布区 。这种长周期的任务要求极其精确的多步推理能力，中间任何一个环节的资源匮乏或环境变故（如遭遇敌对生物攻击、掉入熔岩）都会导致规划链条的断裂 。
第三，无特定目标的高自由度（High Degrees of Freedom）。当以传统模式运行该环境时，系统不会向智能体下达任何强制性的主线剧情或终极目标 。这种高自由度意味着智能体必须拥有内在动机（Intrinsic Motivation）和探索机制，能够根据自身的技能掌握程度和周围的局部环境（例如判断当前处于沙漠还是雪原）来自主设定合适的子目标 。这一特性使得 Minecraft 能够支持长期的代理开发和真正的终身学习研究，为测试 AGI 提供了天然的复杂度梯度（Complexity Gradient）。
在这样的高自由度与长周期环境中，大语言模型（LLMs）正经历着从被动输出文本的“问答模型”向能够主动规划并改变环境状态的“决策智能体”的深刻范式迁移。
## 1.2 国内外研究现状
近年来，基于大型语言模型的Minecraft具身代理研究经历了从单体技能学习到多智能体社会化协同的深刻范式演进，重点围绕算力延迟、逻辑幻觉与通信过载三大瓶颈展开了创新。
在单体智能阶段，开放世界中的探索停滞与灾难性遗忘是首要难题。Voyager (2023/2024) 开创了“代码即技能”的终身学习范式。它通过自动课程机制自发提出探索目标，并调用大模型生成基于 Mineflayer API 的 JavaScript 代码。这些可执行代码被持久化存储于外部向量数据库中，形成了一个可组合、可检索的技能库，极大提升了代理在长期生存中攀升单体科技树的能力 。然而，Voyager 依赖顶层API，绕过了底层像素感知。为此，OmniJARVIS (2024) 实现了视觉-语言-动作 (VLA) 的端到端统一。它创新性地引入“行为编码器”，将连续的行为轨迹量化为离散的“行为 Token”。为缓解大模型直接输出动作带来的高频算力延迟，该框架采用“128步决策周期”，即大模型低频进行宏观思维链推理，轻量级解码器高频执行底层动作，实现了算力与时空的有效解耦 。
随着任务复杂度提升，研究重心向多智能体系统（MAS）转移。在协作初期，自由对话极易导致状态感知误差与逻辑幻觉。VillagerAgent (2024) 引入了软件工程中的图论思想，提出了基于有向无环图 (DAG) 的任务拓扑架构。其中心化的规划器运用零样本推理将复杂指令拆解为严密的DAG图，基于空间与因果依赖向不同智能体精准分发任务，有效降低了团队协作时的逻辑幻觉 。
尽管中心化DAG提升了执行确定性，但在极低延迟的高频互动场景中，频繁的全局重算与通信风暴仍是掣肘。最新进展 MindCraft (2025) 将心智理论 (ToM) 融入底层设计，针对通信过载提出了“分区信息架构”。该系统在内存物理级别隔离了代理的“私有执行状态”（仅依赖确切底层反馈，防范大模型主观臆测导致的污染）与“公共协调状态” 。配合“门控升级机制”，代理在遭遇局部异常时，会综合推演节点关键度与本地恢复成本，只有在协作收益远大于通信延迟代价时才发起对话求助 。这一机制将机械的代码交互升华为具备人类心智特征的战略决策，成功实现了大规模、低延迟的去中心化协作。
综上所述，Minecraft开放世界中的具身智能研究，正沿着“表征的宏微观时空解耦”与“协作的社会化心智觉醒”两条核心主线，加速推动人工智能系统向真实物理世界的通用化部署迈进。
## 1.3 研究意义
本课题面向"大语言模型驱动的开放世界智能体"这一新兴方向,以 Minecraft为统一实验沙盒,系统研究任务表征、决策架构与长期记忆三类核心问题。其研究意义可从理论、实践与工程方法论三个层面加以阐述。
### 1.3.1 理论价值
(1) 拓展"语言模型即决策核"的范式边界。
传统大语言模型研究多聚焦于自然语言理解与生成,而本课题将语言模型置于"感知—决策—执行—反思"的闭环中,系统考察其作为通用决策内核的能力上限与失效模式,为大模型从"问答系统"演化为"具身智能体"提供了可复现的实证支撑与方法论参照。
(2) 引入有向无环图作为长任务的统一表征。 现有工作多采用线性思维链 (Chain-of-Thought) 或单分支树状思维 (Tree of Thoughts)表达任务结构,难以同时承载"目标共享、路径多样、失败回退"三类需求。本文以"状态节点—状态转移边"形式化开放世界中的子目标空间,论证了 DAG在长任务链中相比线性与树状结构具有更高的可达性与鲁棒性,为基于大模型的分层任务规划提供了新的形式化模型。
(3) 借鉴互补学习系统假说构建人工记忆机制。 本课题将认知神经科学中"海马体快速记忆—皮层慢速巩固"的双系统假说迁移到智能体设计,提出"情景记忆—语义记忆—离线巩固"三级人工记忆模型,为人工智能领域如何实现持续学习与经验沉淀给出了一种类脑启发的工程化解释,丰富了类脑计算与认知架构在大模型时代的交叉研究内涵。
### 1.3.2 实践应用价值
(1) 推动开放世界游戏 AI 的研究进展。 Minecraft作为程序生成、长任务链、高自由度并存的开放世界基准,已成为衡量通用人工智能决策能力的重要试金石。本课题给出的智能体框架可直接服务于游戏 NPC行为生成、关卡自动测试、玩家辅助代理等场景,为游戏产业引入"会思考的非玩家角色"提供了一条可工程化的路径。
(2) 可迁移至机器人长程任务规划。 机器人长程操作 (Long-Horizon Manipulation)与本文研究的虚拟世界任务规划在结构上高度同构:二者皆面对部分可观测、动态变化、子目标可达性需反复验证的环境。本文提出的双状态机协同、想象—验证闭环与失败回退机制,可直接迁移到家庭服务机器人、仓储分拣机器人、机械臂装配等具身智能任务,缓解当前机器人系统"一旦失败即陷入死循环"的脆弱性。
(3) 服务自动化运维与科研工作流编排。 在DevOps、数据流水线、生物信息分析等领域,大量任务同样呈现"自然语言目标—子目标DAG—工具调用执行—结果验证"的统一范式。本课题提出的"任务即配置"与"反思—行动—验证"工程模板,可用于构建自动化运维代理、自动实验流水线、自动数据分析工作台等真实生产系统,显著降低专家的重复劳动。
(4) 为大模型应用降本增效。 本文通过"按需检索"取代"上下文塞库",通过"双状态机异步并发"取代"单线程长链路",在保持任务成功率的同时显著降低了 Token消耗与端到端时延,对当下大模型推理成本高企的工业落地具有直接的经济价值。
### 1.3.3 工程方法论价值
(1) 提供一套可复用的智能体工程模板。本课题以依赖倒置与依赖注入容器为核心,将语言模型供应商、知识源、执行环境、记忆策略等组件全部抽象为可替换接口,形成分层清晰、可热插拔的开源框架。后续研究者可在不修改核心逻辑的前提下替换任意子模块,从而加速智能体相关研究的迭代验证。
(2) 形成智能体行为的可解释审计范式。 通过强制思考字段、结构化最终答复协议与决策日志的全链路留痕,智能体的每一次决策、每一次工具调用、每一次失败回退都成为可事后复盘的结构化证据。这一审计范式对大模型走向高风险、强监管的真实应用 (如医疗诊断、金融决策、自动驾驶) 具有重要的方法论参考意义。
综上,本课题既在理论上探索了大语言模型作为开放世界决策内核的可行性边界,又在实践上为多个具身智能与自动化领域提供了可直接迁移的工程模板,兼具学术创新与产业落地双重价值。
## 1.4 本文主要工作与创新点
本文以 Minecraft 这一典型的开放世界沙盒环境作为统一实验载体, 设计并实现了一个由大语言模型驱动、具备长任务规划与持续学习能力的自主智能体系统——DreamCraft。本文的主要工作可概括为以下四方面:
(1) 提出并实现了一个面向开放世界长任务的自主智能体整体框架。 该框架以"领域模型—决策控制—认知服务—基础设施—具身环境"五层架构组织, 通过依赖倒置与依赖注入容器, 将语言模型供应商、知识源、执行环境与记忆策略统一抽象为可替换接口, 形成了一个可扩展、可热插拔的工程化基座。
(2) 设计了一套以"状态节点—状态转移边"为核心的任务领域模型与图操作算法。 该模型将长链决策过程显式化为可达的拓扑有向无环图, 支持分支扩展、路径注入、节点替换、剪枝与回退等运行时操作, 为后续的可行性检查、节点级想象、失败回退与多路径切换提供了统一的形式化基础。
(3) 设计并实现了规划状态机与执行状态机协同的双有限状态机控制架构。 二者通过基于主题路由的异步消息总线解耦通信, 配合"想象—执行—验证—回退"闭环与三次失败熔断机制, 使智能体的复杂决策流程显式化为可枚举、可追溯、可调试的状态迁移图。
(4) 设计并实现了类海马体的离线经验巩固与长期记忆模块。 系统以独立后台协程的形式运行经验蒸馏守护进程, 将历史轨迹自动蒸馏为"程序性技能 + 失败教训 + 抽象任务模式"三类结构化长期记忆, 并通过嵌入索引与近邻去重维护一个可持续增长的记忆库, 使智能体具备"会话间持续学习"的能力。
在上述工作的基础上, 本文形成的主要创新点如下
### 1.4.1 基于反思—行动 (ReAct) 范式的自主工具调用机制
传统智能体倾向于将整个知识库前置塞入提示词, 导致 Token 消耗巨大、信息冗余、关键信息被淹没, 且模型决策过程不可审计。本文提出一种方法将知识检索、技能查询、世界观察、节点导航等能力统一抽象为可被语言模型自主调用的工具接口, 并在每个工具的入参 schema 中强制嵌入"思考字段" (thought), 迫使模型在调用前显式输出推理过程; 同时引入结构化最终答复协议与解析失败回灌的格式自我修复机制, 形成稳定的"思考—检索—观察—再思考"闭环。相比单一提示词基线方案, 在保持任务成功率的同时显著降低 Token 消耗, 并将每一次决策的"为什么这么做"沉淀为可事后复盘的结构化日志, 使智能体行为具备可解释性。
### 1.4.2 基于双有限状态机的协同流程控制架构
以 AutoGPT、BabyAGI 为代表的单循环智能体在长任务中难以同时兼顾"规划"与"执行"两类负载, 一旦中间步骤失败往往陷入混乱无法回退, 且整体流程不可枚举、不可调试。本文将"任务规划状态机" (8 态) 与"动作执行状态机" (5 态) 分别建模为两个并发协程, 通过支持精确等与非阻塞探取的异步消息总线进行通信; 规划侧负责可行性检查、状态想象、粒度判断、任务分解与路径导航, 执行侧负责代码生成、静态校验、异步下发与执行后验证, 两者可对不同节点并行推进。将"规划—执行—验证—反馈—回退"的复杂逻辑显式化为可枚举、可调试的状态迁移图, 在意外失败时由重试—熔断机制自动触发路径回退而非崩溃, 显著提升长任务下的吞吐量与稳定性。
### 1.4.3 将长链决策分解为状态空间上的拓扑有向无环图
思维链 (CoT) 与思维树 (ToT) 等表征只能表达线性或单分支的子目标序列, 无法同时刻画"目标共享、路径多样、失败回退"等真实任务中的关键结构, 也缺乏可被反复验证的状态语义。本文以"状态节点"为图节点、以"状态转移边"为有向边, 将一个高层目标递归地分解为粒度逐级细化的有向无环图; 每个节点同时持有"预想状态快照"与"实际状态快照"两份结构化信息 (含背包、装备、坐标、视野体素、生命/饥饿值、邻近实体等), 并通过"自底向上验证可达终点—不可达即剪枝"的初始化算法保证图始终为可达 DAG; 失败原因被记录于"被阻塞边集合"中, 由规划状态机重新选择替代路径。实现了基于状态空间的可达性判定、节点级反思与多路径替代切换, 使智能体在长链路任务下可以"目标共享、路径多样、失败回退", 显著降低重复扩展并提升整体可达性与鲁棒性。
### 1.4.4 类海马体的离线经验巩固与长期记忆机制
现有大模型智能体多以"上下文窗口 = 全部记忆"的单层结构运作, 无法跨会话沉淀经验, 已踩过的坑往往重复踩, 已掌握的技能也难以稳定复用。本文受认知神经科学"海马体—皮层互补学习系统假说" (CLS, McClelland 1995) 启发, 在系统中引入与主决策循环异步解耦的离线巩固守护协程; 该协程在系统空闲时拉取情景缓冲中的成功/失败轨迹, 通过低成本语言模型蒸馏为"程序性技能 + 失败教训 + 抽象任务模式"三类结构化记忆, 并经嵌入索引、近邻去重合并、LFU 衰减遗忘后写入持久化记忆库; 主循环在每轮决策前以"状态快照 + 当前目标"为查询向量检索 top-k 长期记忆并注入提示词。形成"事实性知识 + 程序性知识 + 情景记忆"的三层认知架构, 使智能体在重复任务上的平均决策步数与 Token 消耗持续下降, 验证"越用越好"的长期学习特性, 同时为类脑认知架构与持续学习研究提供工程化样本。
# 2 相关技术与理论基础
## 2.1 大语言模型与提示工程
### 2.1.1 Transformer 与解码器架构要点
当代主流大语言模型 (Large Language Model, LLM) 以Vaswani等人提出的Transformer为基础架构, 采用多头自注意力 (Multi-Head Self-Attention) 与前馈网络堆叠的解码器结构, 通过自回归方式逐Token生成输出。给定输入序列, 模型以条件概率预测下一Token,整段输出的概率分布可分解为:

### 2.1.2 上下文窗口与 Token 计量
受位置编码与显存预算限制, LLM 的可处理上下文长度存在固定上界 (当前主流商业模型在数万至百万 Token 量级)。文本经字节对编码 (Byte-Pair Encoding, BPE) 或 SentencePiece 分词后转化为 Token 序列, 实际计费与时延均与 Token 数线性相关。本文系统涉及长链路任务, 单次任务对话往往可达数十轮工具调用, 因此 Token 经济性在设计层面具有一阶重要性, 直接驱动了第 4.3 节"按需检索式"提示词架构与第 4.6 节"上下文压缩"机制的设计。
### 2.1.3 提示工程与提示词模板化
提示工程 (Prompt Engineering) 是指通过显式控制模型输入以引导其行为的工程实践。本文采用"角色模板 + 查询模板"的双层 Jinja 模板组合: 角色模板固化模型的身份与全局约束 (如"你是 Minecraft 智能体, 必须严格遵守工具调用格式"), 查询模板按任务类型注入运行时变量, 实现提示词与代码逻辑解耦, 便于调参与 A/B 比较。
### 2.1.4 结构化输出约束
为让模型输出可被下游程序解析, 本文广泛采用 Pydantic 数据模型 + JSON Schema 的双重约束: 在工具调用层强制要求模型按预定义 schema 生成参数, 在最终答复阶段要求其遵循 【Reason】 ... 【Final Answer】 ... 结构化协议; 当解析失败时, 通过将异常以系统消息形式回灌到上下文, 形成"格式自我修复"机制。该机制是第 4.3 节自主工具调用稳定运行的关键。
## 2.2 反思-行动 (ReAct) 范式
(1) 思维链与反思-行动的区别。思维链 (Chain-of-Thought, CoT)通过显式输出中间推理步骤提升模型在复杂推理任务上的准确率, 但其推理过程完全发生在模型内部, 无法与外部世界交互。Yao 等人提出的反思-行动 (Reasoning + Acting, ReAct) 范式在 CoT 基础上引入"行动" (Action) 与"观察" (Observation) 两步, 形成可与外部环境闭环交互的"思考—行动—观察"循环:
其中为模型生成的思考 (Thought),为基于思考决定的动作 (Action, 通常表现为一次工具调用), 为环境对该动作的反馈 (Observation)。本文将 ReAct 作为大模型与游戏世界、知识库交互的统一抽象基座。
(2) 工具调用作为行动的具体形式。 现代大语言模型 (如 GPT、通义、Claude 等) 普遍提供 "Function Calling / Tool Use" 接口, 允许在生成阶段直接输出符合 JSON Schema 约束的工具调用参数。本文将知识检索、技能查询、世界观察、节点导航、代码生成等能力统一封装为可被模型调用的工具, 并在每个工具入参 schema 中强制嵌入 thought字段, 把"思考"显式纳入工具调用本身, 既保留了 ReAct 的可解释性又利用了模型原生的结构化调用通道。
(3) 为何 ReAct 适合作为本文的工具调用基座。 一是它天然支持长链路决策, 与本文 DAG 任务图结构契合; 二是它通过"思考"字段将模型的隐式推理外显化, 便于事后审计与失败溯源; 三是它通过"观察"环节引入外部反馈, 弥补了 LLM 静态知识截止的局限, 使智能体可基于实时状态做出动态决策。
## 2.3 有限状态机与异步并发模型
(1) Mealy 与 Moore 状态机。 有限状态机 (Finite State Machine, FSM) 是一种由有限状态集合、输入字母表、状态转移函数与输出函数组成的离散计算模型, 形式化定义为五元组 , 其中为状态集, 为输入符号集, 为状态转移函数, 为初始状态, 为终止状态集。Mealy 机的输出依赖于"当前状态 + 输入", Moore 机的输出只依赖于"当前状态"。本文第 4.2 节的双状态机均按 Moore 模型组织, 状态本身即决定了当前要执行的子任务 (如"可行性检查"、"状态想象"、"任务分解"等), 便于以状态名为索引绑定提示词与工具集。
(2) 协程与事件循环。 Python 的 asyncio 库基于事件循环 (Event Loop) 调度协程 (Coroutine), 协程通过 await 主动让渡控制权, 由事件循环在 I/O 阻塞时切换到其他就绪任务, 从而以单线程实现高并发。本文系统中, 规划状态机、执行状态机、巩固守护协程均以独立协程的形式运行于同一事件循环, 避免了多线程同步开销, 同时获得了 I/O 密集型任务 (LLM 调用、HTTP 请求) 下的良好吞吐表现。
(3) 异步生产者-消费者与邮箱-总线模型。 当多个协程需要协同时, 通常以"消息队列"作为解耦中介, 形成生产者-消费者模式。本文进一步在普通队列之上构建了一个基于主题路由的"消息总线" (Message Bus): 每条消息携带主题与负载, 订阅方按主题精确等待 (wait_for_topic) 或非阻塞探取 (fetch_topic)。这一模型避免了传统轮询带来的 CPU 浪费与时延抖动, 使规划侧与执行侧在面对"等待对方完成某一具体事件"时具有原生的协程级语义支持。
(4) 在本文中的角色。 第 4.2 节的双状态机协同架构以本节的 Moore FSM 为骨架, 以协程为执行单元, 以主题消息总线为通信介质, 三者共同保证了"规划—执行—验证—回退"流程的可枚举性、可调试性与并发性。
## 2.4 有向无环图与分层任务分解
(1) 有向无环图的形式化定义。 有向无环图 (Directed Acyclic Graph, DAG) 是一个二元组, 其中为节点集, 为有向边集, 且不存在任何节点可经由若干条有向边回到自身。DAG 的关键性质包括: 拓扑排序的存在性 (节点可被线性化使所有边指向后继)、可达性闭包的可计算性、以及天然支持"多源—多汇—多路径"的层次结构表达。
(2) 拓扑排序、路径搜索与剪枝。 拓扑排序通常以 Kahn 算法或基于深度优先搜索的逆后序实现, 时间复杂度。路径搜索可在拓扑序基础上以动态规划求得最短/最优路径; 剪枝则是在路径搜索过程中按一定可行性判据 (如不可达终点) 截断分支。本文第 4.1 节的"自底向上验证可达终点—不可达即剪枝"算法即是该范式的具体应用: 从终点反向标记可达节点, 凡未被标记者直接剪除, 保证图始终是可达 DAG。
(3) 分层任务网络与本文任务图的关系。 经典分层任务网络 (Hierarchical Task Network, HTN) 将复合任务递归分解为子任务直至原子动作, 与行为树 (Behavior Tree) 一道构成传统 AI 规划的两大范式。本文的任务图保留了 HTN 的"递归分解"思想, 但在表征上不再使用人工预定义的方法库 (method library), 而是由大语言模型在运行时动态生成子节点, 同时以 DAG 替代 HTN 的"任务树", 以容纳"多目标共享子任务"与"失败回退切换替代路径"两类传统 HTN 难以表达的结构。
(4) "状态节点—状态转移边"形式化定义。本文将任务图节点定义为四元组, 其中为节点标识与自然语言描述, 为规划阶段由模型生成的"预想状态快照",  为执行后采集的"实际状态快照"; 状态转移边定义为无序对上的有向操作, 携带"可执行代码"、"是否被阻塞"、"失败原因"等运行时元数据。该形式化是第 4.1 节算法描述的基础术语。
## 2.5 检索增强生成与向量化知识库
(1) 文本嵌入与向量空间。 文本嵌入 (Text Embedding) 通过预训练编码器将自然语言映射为定长稠密向量, 使语义相近的文本在向量空间中距离更近。本文采用与主流商业 API 兼容的1536维嵌入模型, 对知识文档块、技能说明、长期记忆条目分别建立向量索引。
(2) 距离度量。衡量两段文本相似度的常用度量包括欧氏距离 (L2) 与余弦相似度 (cosine):
当嵌入向量经过 L2 归一化后, 余弦相似度与欧氏距离单调等价, 但余弦相似度对向量模长不敏感, 更适用于文本检索场景, 因此本文以余弦相似度为默认度量。
(3) 近似最近邻索引。 当向量库规模达到万级以上, 精确最近邻搜索 (Brute-Force kNN) 的时间复杂度难以满足实时检索需求, 业界普遍采用近似最近邻 (Approximate Nearest Neighbor, ANN) 索引, 例如 HNSW (Hierarchical Navigable Small World) 与 IVF-PQ (Inverted File + Product Quantization)。前者在多层导航小世界图上做贪心跳转, 检索时间近似，精度高、易增量更新; 后者通过聚类与乘积量化压缩向量, 适合超大规模。本文综合考虑增量写入需求与库容量, 选用 HNSW 作为长期记忆与知识库的统一索引。
(4) 检索增强生成的必要性。 检索增强生成 (Retrieval-Augmented Generation, RAG) 是指在生成前先检索相关上下文并注入提示词的范式。相比"把整个知识库前置塞入提示词"的朴素方案, RAG 在 Token 经济性、信息相关性、知识可更新性三方面均具有显著优势。这一点直接对应本文创新点 1 中"按需检索取代上下文塞库"的设计动机。
## 2.6 互补学习系统与人工记忆巩固
(1) 互补学习系统假说的生物学基础。 McClelland、McNaughton 与 O'Reilly 于 1995 年提出互补学习系统假说 (Complementary Learning Systems, CLS), 认为哺乳动物大脑通过"海马体—皮层"两套互补系统协同实现快速学习与长期巩固: 海马体负责快速、稀疏、特征独立的情景式记忆 (Episodic Memory), 在事件发生时迅速建立稀疏编码; 大脑皮层负责慢速、稠密、特征交叠的语义式记忆 (Semantic Memory), 通过反复学习抽象出泛化模式。睡眠或静息期间, 海马体中的短期痕迹通过"重放" (Replay) 缓慢迁移到皮层, 完成长期巩固并避免新旧知识相互干扰 (灾难性遗忘)。
(2) 在人工智能中的对应物。 CLS 思想在深度学习领域演化出"经验回放" (Experience Replay)、双网络架构 (DQN 中的 online network 与 target network) 等机制。在大模型智能体领域, Generative Agents 中的"记忆流 + 重要性评分 + 反思"流程、MemGPT 中"主存—虚拟存"分层内存、Voyager 的"自我编程式技能库膨胀", 都可视为对 CLS 不同侧面的工程化模仿。
(3) 情景记忆与语义记忆的工程化分离。 在工程实践中, 情景记忆通常以原始轨迹/对话日志的形式存在, 检索粒度细但噪声大; 语义记忆则以结构化、去重、可索引的形式存在, 检索粒度粗但信噪比高。两者通过"蒸馏"流程相联: 情景记忆作为输入, 经语言模型抽取出结构化、可复用的语义记忆。本文第 4.7 节的离线巩固守护协程即是这一思想的具体实现。
(4) 在本文中的角色。 本节构成第 4.7 节"类海马体记忆模块"的理论铺垫: 主决策循环对应海马体的快速学习, 离线巩固协程对应睡眠期重放, 长期记忆库对应皮层语义网络, 三者通过事件驱动协作形成"事实性知识 + 程序性知识 + 情景记忆"的三层认知架构。
## 2.7 Minecraft 仿真平台与具身交互层
(1) Minecraft 与开放世界基准。Minecraft 是一款基于体素 (Voxel) 的开放世界沙盒游戏, 其程序生成地形、深度合成依赖、长任务结构、多模态交互特性使其成为开放世界智能体研究的事实基准, 衍生出 MineRL (强化学习数据集与竞赛)、MineDojo (任务库与文本-视频知识库)、Voyager (终身学习智能体) 等代表性研究平台。本文采用 Minecraft Java 1.20.x 作为实验环境, 兼容主流社区机器人开发框架。
(2) Minecraft Java 版协议与机器人开发框架。Minecraft Java 版以 TCP 长连接承载二进制数据包, 服务器维护世界状态权威。Node.js 生态中的 Mineflayer 框架对协议进行了完整封装, 暴露出"挖掘""移动""合成""攻击"等高层动作 API, 并以事件回调形式上报世界变化。本文以 Mineflayer 为基础, 在其上构建一层 Express HTTP 服务, 将动作 API 与世界观察接口重新封装为语言无关的 REST 端点。
(3) HTTP 服务作为 Python 与游戏世界的具身桥梁。Python 主程序与 Node.js 端通过本地 HTTP 通信, 主要提供启动会话、世界观察、代码下发、关闭会话四类动作; 观察接口在 Node 端预先完成体素抽稀、实体距离归一化、背包序列化等预处理, 把原始 3D 世界投影为语言模型可读的结构化状态快照。这种"跨语言、跨进程"的桥接架构既复用了 Mineflayer 已有的协议实现, 又使主决策逻辑得以以 Python 异步生态独立演化, 二者通过明确的 REST 契约解耦。
(4) 代码沙箱。为安全执行语言模型生成的代码, Node 端将其包裹为立即执行的异步函数 (IIFE), 在 vm 模块下隔离运行, 配合超时与异常捕获形成沙盒边界。


# 3 系统总体设计
## 3.1 系统目标与需求分析
### 3.1.1 Transformer 总体目标
构建一个以大语言模型为决策核心、以 Minecraft 为具身环境的自主智能体系统, 使其能够仅凭一句自然语言目标 (如 "获得一把铁镐""击败末影龙"), 自主完成从任务理解、子目标分解、代码合成、具身执行、状态验证到经验沉淀的完整闭环, 并在多次会话间持续累积可复用经验。该目标既覆盖了"单次任务能完成"的可行性指标, 也强调了"长期使用越来越好"的持续学习特性。
### 3.1.2 功能需求
按主用例划分, 系统需提供以下五类核心功能:
(1) 自然语言任务接入 接受用户的自然语言目标作为唯一输入, 不假设其携带任何结构化字段, 由系统内部完成解析与子目标抽取。
(2) 任务自主分解 在大语言模型协助下将高层目标递归分解为有向无环图形式的子目标空间, 并具备根据世界状态动态调整分解粒度的能力。
(3) 具身自主执行 针对每个原子节点自动合成 JavaScript 动作代码, 经语法校验后下发到游戏环境异步执行, 并采集执行后的世界快照。
(4) 执行结果自我验证 对比"预想状态快照"与"实际状态快照", 形成节点级反思; 失败时按既定策略回退路径而非直接终止。
(5) 经验跨会话沉淀 在系统空闲时对历史轨迹进行结构化蒸馏, 生成程序性技能、失败教训与抽象任务模式三类长期记忆, 供后续任务复用。
### 3.1.3 非功能需求
围绕开放世界长任务的工程特点, 系统在以下五个非功能维度提出明确指标:
(1) Token 经济性 优先采用按需检索而非全量上下文注入, 单次决策的平均 Token 消耗应显著低于"提示词塞库"基线。
(2) 可解释性 每一次模型决策、工具调用与状态转移均留有结构化日志, 支持事后逐步回放与审计。
(3) 可恢复性 任意单节点的执行失败不得导致系统崩溃; 系统应能依靠回退策略自动选择替代路径或终止上报失败原因。
(4) 可扩展性 大语言模型供应商、知识来源、向量索引实现、执行环境等四类组件均可在不修改业务逻辑的前提下替换。
(5) 长期可用性 系统在持续运行下应表现出"经验越积越多, 重复任务越做越快"的单调改进趋势, 而非性能漂移。
## 3.2 总体架构设计
### 3.2.1 分层架构概述
为同时满足上述功能与非功能需求, DreamCraft 采用自上而下五层架构: 领域模型层、服务层、基础设施层、环境层、横切关注层。每一层只依赖其下层提供的抽象接口, 上层并不知道下层的具体实现, 从而保证关键替换点的局部性。
(1) 领域模型层 (Domain Layer) 定义状态节点、状态转移边、任务图、状态快照、技能、长期记忆条目等核心领域对象。该层不依赖任何外部框架, 只用纯 Python 数据模型描述"系统在谈论什么"。
(2) 基础设施层 (Infrastructure Layer) 容纳具体的实现细节, 包括语言模型客户端 (OpenAI/通义兼容接口)、文本嵌入与向量索引、知识文档库、技能库仓储、提示词仓储、工具注册表、具身环境客户端等。该层是系统中唯一与外部 SDK / 第三方库直接交互的区域。
(3) 服务层 (Service Layer) 主要提供四类高层服务、两种状态机以及消息传递，包括：语言模型推理服务、知识检索服务、任务管理服务、记忆巩固服务、规划状态机、执行状态机、异步消息总线、任务上下文管理器等运行时编排逻辑。对外呈现以"任务对象 → 结构化结果"为契约的统一调用入口, 内部聚合多个基础设施组件。
(4) 环境层 (Environment Layer) 系统外部的真实世界, 包括基于 Node.js 的游戏交互服务、Minecraft 服务器以及 Microsoft/Mojang 账号鉴权服务。系统通过 HTTP 接口与该层通信, 不在 Python 端假设任何游戏协议细节。
(5) 横切关注层（Cross-cutting Layer） 跨越多个模块的功能需求，它们在应用中横向分散，提供与项目无关的基础服务（utils），包括：Grep文件正则搜索工具、日志格式化输出工具、子进程创建与管理工具。他们不涉及项目的具体内容，仅仅作为横切提供工具方法。
### 3.2.2 依赖倒置与依赖注入容器
为保证上述"局部替换"特性, 系统在工程上严格执行依赖倒置原则 (Dependency Inversion Principle, DIP): 高层模块不依赖低层模块, 二者都依赖于抽象。具体到代码层面,服务层仅持有协议 (Protocol) 接口, 不直接 import 任何具体实现; 实现类则在启动时由统一的依赖注入容器 (Container) 装配并交付，该容器以纯函数式风格组织, 不引入额外的运行时反射开销，共负责以下内容： 
(1) 在程序启动时按配置选择具体实现 例如选择 OpenAI 还是通义嵌入;
(2) 维护单例与作用域 例如向量索引在进程内仅初始化一次;
(3) 在测试场景下注入模拟实现, 便于单元测试与回归验证。

## 3.4 数据流与控制流设计
### 3.4.1 端到端数据流
一次完整任务的数据在系统中的流动路径如下: 用户的自然语言目标进入决策控制层后, 首先被任务管理服务包装为根任务图节点 (起点 = 当前世界状态, 终点 = 目标状态); 规划状态机在大语言模型协助下不断对终点节点做可行性检查与状态想象, 并通过任务分解将其展开为子节点, 最终形成可达 DAG; 路径导航选定当前要执行的节点后, 执行状态机调用代码生成服务合成 JavaScript 代码; 代码经静态语法校验后通过具身环境客户端下发到 Node.js 服务并由 Minecraft 服务器执行; 执行完毕的世界快照沿原路径返回, 由执行状态机比对预想快照, 形成验证结果; 若成功则推进到下一节点, 若失败则将失败原因写入被阻塞边集合并触发路径调整。
### 3.4.2 并发双循环控制流
系统在控制流层面由两个个独立的异步协程并发运行:规划状态机协程负责"思考下一步做什么"; 执行状态机协程负责"把当前节点真正做完";。两者通过同一个基于主题路由的异步消息总线通信, 既保证彼此完全解耦, 又允许在需要时精确等待特定事件 (例如执行侧等待规划侧下发"开始执行 X 节点"的消息)。
这种"思考—执行"双协程并行的设计带来三方面收益: 一是规划侧可以在执行侧仍在执行当前节点时, 提前对后继节点做可行性检查与状态想象, 隐藏 LLM 推理时延; 二是动作协程独立运行, 不会与主决策循环争抢 CPU 与上下文窗口; 三是任一协程发生异常都被局限在其状态机内部, 不会污染其他协程的状态机。
## 3.5 技术选型
### 3.5.1 选型原则与优势
技术选型围绕三条原则展开: ① 生态成熟——优先选用社区活跃、文档完善的库, 降低维护成本; ② 可替换——避免与特定厂商深度耦合, 关键能力 (LLM、嵌入、向量库) 均通过协议接口接入; ③ 异步友好——所有 I/O 密集型组件均需提供原生异步 API 或可包装的异步接口, 以契合 asyncio 事件循环。
### 3.5.2 与既有方案的差异
与 Voyager 等以 gym.Env 为统一运行框架的方案不同, 本文未直接采用大型 Agent 框架, 而是仅借鉴其工具调用与 ReAct 的范式思想, 在核心层使用自研轻量协议接口与依赖注入容器。这一选择换来三方面优势: ① 避免被框架版本升级所绑架, 长期可维护性更好; ② 协议接口的语义更贴近本文领域模型 (任务图、状态节点、状态快照), 减少了概念转译成本; ③ 调试链路更短, 任何异常可直接在自有代码中复现, 不需深入框架内部。
# 4 关键模块的详细设计与实现
## 4.1 任务领域模型: 状态节点与状态空间图
### 4.1.1 设计动机
开放世界任务的两大特征——"目标可达路径不唯一"与"中间步骤可能失败"——决定了任务表征不能采用线性链或单分支树, 必须支持节点共享、分支合并、失败回退与替代路径切换。为此, 本文以有向无环图 (DAG) 作为任务的核心表征, 并在节点层面引入"预想—实际"双轨快照, 使每一次状态转移都成为可验证的最小回路。
### 4.1.2 状态快照: 世界状态的统一表征
状态快照 (State Snapshot) 是语言模型系统中描述"世界长什么样"的标准化数据结构, 也是可达性判定、状态想象与执行后验证三类决策任务的统一输入输出。本质上是从完整的观察（Observation）数据模型中提炼出的适合LLM阅读的关键信息。每个快照按结构化字段组织以下信息: 背包内容 (物品 ID + 数量)、装备槽 (主手/副手/护甲)、当前坐标与朝向、视野范围内的体素列表 (类型 + 相对坐标, 已经过抽稀)、生命值与饥饿值、附近实体的类型与归一化距离, 以及一段自然语言"环境摘要"用于上下文注入。所有字段均通过 Pydantic 数据模型校验, 保证序列化前后无歧义。
