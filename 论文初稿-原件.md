## 摘要

近年来, 以大语言模型 (Large Language Model, LLM) 为核心驱动的具身智能体在开放世界中的自主决策与任务规划能力取得了显著进展。然而, 当面向 Minecraft 这类具有海量物品、深度合成依赖、长链路任务结构的开放世界沙盒环境时, 现有方案普遍存在三方面问题: 一是 LLM 在长链决策中容易出现"上下文遗忘"和"幻觉跳步", 难以稳定完成多阶段任务; 二是大多数智能体采用"一次性把所有信息塞进 prompt"的被动方式, 信息冗余且 token 消耗巨大; 三是任务执行流程缺乏严格的状态管理, 失败后难以回退、复用和反思。

针对上述问题, 本文设计并实现了一个面向 Minecraft 的自主智能体系统 DreamCraft。本文的主要工作和创新点包括:

1. **基于 ReAct 范式的 AI 自主工具调用机制**。系统将 Wiki 知识库、Skill 技能库、世界观察、节点查询等能力统一封装为 LangChain Tool, 由 LLM 通过 `thought` 字段进行自我反思后, 按需主动调用, 形成"思考-调用-观察-再思考"的闭环, 显著降低了一次性 prompt 的体积与 token 消耗。
2. **基于双状态机的协同流程控制架构**。系统由 `QuestOrchestrator` (规划器) 与 `QuestExecutor` (执行器) 两个并发状态机协同工作, 通过 `MessageBus` 进行异步消息通信, 将"规划-执行-验证-反馈-回退"的复杂逻辑显式化为可枚举、可调试的状态迁移图, 提升了系统的可解释性与稳定性。
3. **将长链决策链分解为拓扑有向无环图 (Topological DAG)**。系统以 `Waypoint` (路径点) 为节点、`Edge` 为边, 将一个高层目标 (如"击败末影龙") 递归地分解为粒度逐级细化的 DAG; 每个节点同时持有"想象快照 (imaginated_snapshot)"与"实际快照 (actual_snapshot)", 实现了基于状态空间的可达性判定与失败回退。
4. **类海马体的后台经验提炼与记忆机制 (设计与展望)**。本文进一步提出一种受生物海马体启发的离线记忆巩固模块, 在主决策循环空闲时对历史轨迹进行总结、聚类与索引, 形成可被未来任务检索和复用的长期经验。该部分目前以接口预留与设计原型的形式给出, 作为未来工作的重点扩展方向。

实验部分以 Minecraft Java 版为测试环境, 通过 mineflayer 框架与 Python 主程序进行 HTTP 通信, 在多组任务 (从"获取木板"到"获取铁镐"再到长链路的"建立基础生存基地") 上验证了系统的可行性。结果表明, DreamCraft 在任务成功率、token 使用效率、长任务稳定性等指标上均优于以单一 prompt 直接驱动 LLM 的基线方案。

**关键词**: 大语言模型; 自主智能体; Minecraft; ReAct; 状态机; 有向无环图; 任务规划; 经验记忆

## Abstract

In recent years, embodied agents driven by Large Language Models (LLMs) have made significant progress in autonomous decision-making and task planning in open-world environments. However, when applied to sandbox environments such as Minecraft — which contain massive item spaces, deep crafting dependencies, and long-horizon task structures — existing approaches still suffer from three major issues: long-chain context forgetting and hallucinated step-skipping; passive "one-shot prompt" information injection that is highly redundant and token-expensive; and lack of explicit state management which prevents principled rollback, reuse and reflection on failure.

To address these problems, this thesis designs and implements **DreamCraft**, an autonomous LLM-driven agent system for Minecraft. The main contributions are:

1. A **ReAct-style autonomous tool-invocation mechanism** that wraps the Wiki knowledge base, skill library, world observation and graph navigation as unified LangChain tools, allowing the LLM to retrieve information on demand via a self-reflective `thought` field, drastically reducing one-shot prompt size and token cost.
2. A **dual finite-state-machine cooperative control architecture**, in which a `QuestOrchestrator` and a `QuestExecutor` run concurrently and communicate through an asynchronous `MessageBus`, making the planning-execution-verification-rollback flow explicit, debuggable and stable.
3. A **topological DAG decomposition of long decision chains**, where each `Waypoint` carries both an *imaginated* and an *actual* world snapshot, enabling reachability checks and principled rollback in state space.
4. A **hippocampus-inspired offline memory consolidation module** (designed and to be implemented), which distils past trajectories into reusable long-term experience during the agent's idle time.

Experiments on the Minecraft Java Edition (driven via the mineflayer Node.js framework) on tasks ranging from short crafting goals to long-horizon survival objectives show that DreamCraft outperforms single-prompt LLM baselines in success rate, token efficiency and long-task stability.

**Keywords**: Large Language Model; Autonomous Agent; Minecraft; ReAct; Finite State Machine; Directed Acyclic Graph; Task Planning; Experience Memory

### 第4章 关键模块的详细设计与实现 (论文核心章)

#### 4.1 任务领域模型: Waypoint / Edge / Quest

- **核心数据结构**: `Waypoint` (含 name/description/imaginated_snapshot/actual_snapshot/extra_info/next/prev/ind), `Edge` (基于 frozenset 的无序对哈希), `Quest` (持有 origin/target/waypoints/exec_path/blocked_edges/exec_history)。
- **图操作算法**: `branch_to`, `insert_after`, `insert_replace_after`, `insert_between`, `inject_between`, `link_between`, `prune` (包括"自底向上验证可达终点-不可达即剪枝"的 `init` 算法)。
- **可达性与回退**: `blocked_edges` 字典记录失败原因, `exec_path` 在执行失败时被截断回退。
- **路径快照表征**: 每个 `Snapshot` 包含 inventory、equipment、position、voxels、food/health 等字段, 是 Imaginate / Verify / Feasibility 三类任务的统一输入。

> **创新点 3 重点段落**: 详细论证"为什么用 DAG 而不是单链或树": 多路径并存以表达"得到铁镐"既可以挖也可以交易; 节点共享子目标可避免重复扩展; 拓扑结构支持失败回退与替代路径切换。

#### 4.2 双状态机协同控制: Orchestrator + Executor

- **OrchestratorState** (8 态): `INIT → CHECK_FEASIBILITY → IMAGINATE → CHECK_GRANULARITY → EXPAND → NAVIGATE → WAIT_FOR_EXECUTOR → SUCCESS`, 给出完整状态迁移图与每个 handler 的语义。
  - `CHECK_FEASIBILITY`: LLM 判定从 current 到 next 是否可由现有技能直接达成。
  - `IMAGINATE`: 让 LLM"想象"下一节点完成后的世界状态, 用于后续 `VERIFY` 比对 (论文重点突出此"想象-实际"双轨对照机制, 是节点级反思的关键)。
  - `CHECK_GRANULARITY`: 判断节点是否已足够原子, 不再需要分解。
  - `EXPAND`: 调用 `ExpandPathTask` 让 LLM 输出子目标列表并通过 `expand_between` 注入 DAG。
  - `NAVIGATE`: 在已展开的 DAG 中选择下一个可行替代路径。
- **ExecutorState** (5 态): `INIT → WAIT → GENERATE_CODE → VERIFY → SUCCESS`, 强调:
  - 代码生成失败的"3 次熔断"机制 (`fail_records > 3` 时上报失败)。
  - 生成代码 → `KnowledgeService.inject_dependencies` 自动拼接技能依赖 → `esprima` 静态语法校验 → `mc.execute` 异步下发。
  - `VERIFY` 阶段对比 `imaginated_snapshot` 与 `actual_snapshot` 形成"想象-验证"闭环。
- **MessageBus 与 Mailbox**: 描述 `wait_for_topic` / `fetch_topic` / `emit_to` 的实现, 论证 asyncio.Future + 主题路由如何避免轮询并支持精确等待。

> **创新点 2 重点段落**: 强调"双状态机 + 异步邮箱"与传统单循环 Agent (如 AutoGPT) 的差异, 说明这种设计如何把"思考"和"执行"解耦, 让二者可以并行 (Orchestrator 提前规划下一节点, Executor 仍在执行当前节点)。

#### 4.3 LLM 服务层与 ReAct 自主工具调用

- **统一执行入口** `LLMService.execute(task)`: 任务以 `BaseTask` 的 Pydantic 子类组织, 自带 `extra_tools` / `parser` / `max_iterations` / `enable_context_compression` 等元信息, 实现"任务即配置"。
- **ReAct 主循环** `LLMService.react`:
  - 强制 `thought` 字段: 通过 `@need_thought` 装饰器在每个工具的 schema 上注入必填的 `thought` 参数, 迫使 LLM 在每次调用前进行自我反思 (论文重点说明这是稳定 ReAct 的关键技巧)。
  - **并发工具调用**: 单轮内多个 tool_call 通过 `asyncio.gather` 并发执行, 显著降低长链路 latency。
  - **Final Answer 协议**: LLM 必须以 `【Reason】 ... 【Final Answer】 ...` 结束, parser 校验失败会被回退为 HumanMessage 重试, 形成"格式自我修复"。
- **任务模板系统** (`PromptRepo` + `prompts/*.md`): 每个任务有 role / query 双模板, 通过 `BaseTask.name` 自动映射, 实现 prompt 与代码解耦。

> **创新点 1 重点段落**: 详细分析"AI 自主调用工具" 相比传统"一次性 Prompt"的优势:
> - **token 经济性**: 不需要把整本 Wiki 塞进上下文。
> - **信息相关性**: LLM 在 thought 中明确"还差什么", 才发起检索, 检索结果就是问题答案。
> - **可解释性**: thought 字段相当于一份带签名的决策日志。
> 同时给出 token 使用对比表 (基线 vs DreamCraft) 作为支撑数据。

#### 4.4 知识层: Wiki RAG 与 Skill 库

- **Wiki 知识库**:
  - `query_wiki`: 基于通义嵌入的语义检索, 返回 L2 距离辅助 LLM 自我评估检索质量。
  - `grep_wiki_files`: 关键字 + heading_hierarchy 的 ripgrep 风格全局检索。
  - `read_wiki_section`: 按章节 (含子章节) 精读, 三者构成"模糊 → 全局 → 精读"的三级检索体系。
- **Skill 库**: 基于 JS 函数描述 + 嵌入向量的技能记忆, 支持 `query_skill`, 并通过 AST/正则解析自动维护技能间的依赖关系 (`inject_dependencies` 在生成代码时自动拼接所需技能)。
- 与第4.7节的"经验记忆"形成"事实性知识 (Wiki) + 程序性知识 (Skill) + 情景记忆 (经验)"的三层认知架构。

#### 4.5 执行环境: Mineflayer 通信桥

- **Python 端 `Agent`**: 基于 httpx 的异步 HTTP 客户端, 提供 `start / observe / execute / close` 四个动作。
- **Node.js 端 `mineflayer_server/index.js`**: Express 服务封装 mineflayer Bot, 提供同名 REST 接口, 内置 voxel 抽稀、entity 距离归一化、inventory 序列化等观察预处理。
- **Azure 登录支持**: 通过 minecraft-launcher-lib 实现正版账号 OAuth 流程 (可选, 由 settings.azure_login 控制)。
- **代码沙箱**: 用 `(async () => { ... })()` 包裹 LLM 生成代码并在 Node 端 vm 模块下执行, 配合 300s 超时与异常捕获形成隔离层。

#### 4.6 上下文压缩与 token 控制

- 介绍 `summary` 工具与 4000 字符阈值触发的"摘要替换历史"机制: 当对话累积超过阈值时, 用 LLM 自总结的 `【全局摘要记忆】` 替换早期 ToolMessage, 仅保留最新一轮工具调用上下文。
- 给出压缩前后 token 数对比, 说明这是 ReAct 长循环不爆炸的关键工程技巧。

#### 4.7 类海马体经验记忆模块 (设计与原型, 创新点 4)

> 该节明确说明: **该模块属于本文的设计与展望部分, 当前版本以接口预留、原型实现的形式呈现, 完整实现作为未来工作。**

- **生物学动机**: 简述海马体 → 皮层的离线巩固假说 (Replay / Sharp-wave Ripples) 与人工 Agent 长期记忆的类比。
- **架构设计**:
  - **Episodic Buffer**: 主循环每完成一个 Quest, 将完整轨迹 (Waypoint DAG + Snapshot 序列 + tool_call 日志 + 成败标签) 写入情景缓冲。
  - **Consolidator (后台守护协程)**: 在 `MessageBus` 空闲时间触发, 调用 LLM 对成败轨迹分别进行总结, 提炼为三类条目:
    1. **新技能 (Skill)** — 写入 SkillRepo 的可执行 JS 函数。
    2. **失败教训 (Lesson)** — 用于 future Feasibility / Generate Code 的 "negative few-shot"。
    3. **抽象模式 (Schema)** — 跨任务复用的子 DAG 模板。
  - **Retriever**: 在 `LLMService.execute` 之前, 以当前 Snapshot 与 target 为 query 检索经验, 注入到 prompt 的 `extra` 段。
- **核心算法草图** (伪代码): 给出 Replay-based 总结的算法流程, 强调"压缩 + 复述 + 索引"三步循环。
- **预期收益**: 任务越做越快、错误越来越少、token 消耗逐次下降。
- **遗留问题**: 经验冗余去重、错误经验的污染 (catastrophic forgetting)、多任务公平采样。
